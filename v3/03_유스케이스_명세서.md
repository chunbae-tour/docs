## 0. 확정 구현 범위 요약

실제 구현 기준 액터별 유스케이스 전체 범위. 상세 명세는 Section 4 참조.

| Actor | 확정 유스케이스 |
| --- | --- |
| `Guest` (비회원) | 관광지·전통시장·축제·가게 공개 조회, 지도 마커 탐색, 통합 검색, 인기 검색어 조회, 축제 캘린더 조회, 게시글 조회, FAQ 조회 |
| `USER` | 회원가입, 로그인(일반·소셜), 로그아웃, 마이페이지 조회, 찜(관광지·전통시장·축제), 관광지 리뷰 작성, 키워드 검색, 길찾기(카카오맵 연결), 게시글·댓글 작성, 신고, 채팅방 개설·참여 신청·참여, 실시간 채팅·번역, 동행 생성·완료, 동행 리뷰 작성, 엽전 충전, 결제 내역 조회, 상품 구매, QR 결제, 환불 신청, 알림 조회·읽음, 고객센터 상담, 상인 신청 |
| `MERCHANT` | 가게 등록·수정(이미지·공지 포함), 메뉴 관리, QR 코드 발급, QR 결제 승인·거절, 정산 내역 조회·출금 신청, 광고 신청 |
| `ADMIN` | 회원 관리(경고·정지·탈퇴), 상인 인증 처리, 신고 처리(콘텐츠 숨김·제재), 관광지·전통시장·축제 데이터 수집·관리, 광고 승인·거절, 배너·FAQ·상품 관리, 환불 처리, 정산 관리, 고객센터 상담 관리, 감사 로그 |
| 외부 시스템 | PortOne(PG 결제), Google Translation API(채팅 번역), Kakao OAuth·딥링크, Tour API·공공데이터포털(축제·관광지 수집), S3(이미지 저장), Redis(Geospatial·ZSet·Pub/Sub), MySQL |

### 공통 예외 처리 원칙

- 인증 토큰 없이 보호 API 접근 → `AUTH_006` 반환
- 권한 부족(role 불일치) → `AUTH_007` 반환
- 제재 사용자 → 커뮤니티·리뷰·댓글 등 도메인별 활동 차단
- QR 결제 → 만료·중복 대기·본인 가게 결제·잔액 부족 각각 별도 에러 처리
- 분산 락(Redisson) → 엽전 차감·QR 결제 동시성 보호
- Redis Cluster → multi-key 명령 hash slot 제약 준수

## 1. 액터 정의

| 액터 | 설명 |
| --- | --- |
| `Guest` | 비회원 (로그인 없이 공개 정보 조회 가능) |
| `USER` | 일반 로그인 사용자 (내·외국인 여행자). 기본 role. |
| `MERCHANT` | USER가 상인 신청 후 ADMIN 승인을 받으면 role이 MERCHANT로 전환된다. 동일 계정에서 상인 전용 기능(가게 관리·메뉴·QR·정산·광고)이 활성화되며, USER 기능 일부도 병행 사용 가능하다. 로그인 시 별도 진입 경로(`/merchant`)로 접속한다. |
| `ADMIN` | 서비스 운영 관리자. 별도 진입 경로(`/admin`)로 접속한다. |
| `System` | 내부 자동화 처리 (Redis, 알림, 번역, 스케줄러 등) |

**상인 권한 전환 흐름**

```
일반 회원가입 → USER 계정 생성 → 상인 등록 신청 → 관리자 승인 → role이 MERCHANT로 변경 → 상인 페이지 접근 가능
```
