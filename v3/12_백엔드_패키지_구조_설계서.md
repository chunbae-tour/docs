## 0. 최종 패키지 구조 보강

`src/main/java/com/chunbaetour/domain` 직속 디렉터리는 다음 21개다(전수 확인).

| 패키지 | 역할 |
| --- | --- |
| admin | 관리자 전용 컨트롤러/서비스 — 내부에 8개 하위 도메인(audit, banner, certification, dashboard, market, place, shop, user)을 둔 묶음 패키지. 엔티티는 안 가짐(다른 도메인 엔티티를 관리자 권한으로 조작) |
| auth | 인증, 회원 계정(Account), OAuth(카카오/네이버), JWT, Security 필터/설정, 프로필 이미지, 정지 만료 스케줄러 — **Security/JWT/OAuth가 모두 이 패키지 안에 있다** |
| banner | 배너 엔티티/리포지토리/상태 enum만 — 컨트롤러·서비스는 `admin.banner`에 있음(관리자만 CRUD하는 리소스라 일반 도메인 서비스가 없음) |
| chat | 채팅방, 메시지, 참여 신청, WebSocket 설정(`StompChannelInterceptor`, `WebSocketConfig`), 채팅 파일 S3/로컬 스토리지 |
| common | 공통 응답/예외/엔티티/설정/레이트리밋/감사로그/시크릿검증/유틸 — **사실상 이 프로젝트의 "global" 패키지** (별도 `global` 패키지는 없음) |
| community | 자유게시글(`free`), 동행 모집 게시글(`companion`), 댓글(`comment`), 공통 게시글 조회(`common`)로 4분할 |
| companionreview | 동행(Companion) 생애주기 + 동행 리뷰 — `community.companion`(모집 게시글)과는 별개 패키지(이름이 비슷해 혼동 주의) |
| cs | FAQ + 1:1 상담방/메시지 — **구 "support" 도메인이 여기로 통합됐다.** AI FAQ를 위한 별도 `ai` 패키지는 없음(FaqService가 cs 안에 있음) |
| festival | 축제, 캘린더, 외부 API(TourAPI) 수집, 축제 찜 |
| like | 범용 찜(폴리모픽 대상) — `place.service.PlaceLikeService`(관광지 전용 찜 응답/캐시 갱신)와 역할이 겹쳐 보이는 지점이 있으나, `like` 패키지는 대상 타입 공용 처리, `place`는 관광지 전용 응답 포맷팅을 맡는 분리 구조다 |
| market | 전통시장 — `place`와 별개 최상위 패키지지만 에러코드 prefix는 `PLACE_*`를 공유한다(설계 의도, [11_공통_에러코드_설계서] 참조) |
| merchant | 상인 신청/인증 신청 처리(신청 단계만). 가게 실체는 `shop` |
| notification | 알림 — Redis Pub/Sub로 다중 서버 개인 알림 전달 |
| payment | 충전, QR 결제, 환불, PortOne 웹훅, 멱등성, 정산 잠금 설정. 스케줄러가 4개(만료/재시도/리컨실/한도)로 가장 많음 |
| place | 관광지, 검색, 추천, 리뷰, 찜, 길찾기, 지오코딩(역지오코딩 포함), Kakao/TourAPI 연동, MySQL Spatial 함수 기반 근접 탐색 |
| report | 신고 + 제재(Sanction) — 대상 타입별 `validator`/`listener`로 폴리모픽 신고 처리, `interceptor`로 제재 계정 쓰기 차단 |
| search | 통합 검색, 인기/최근 검색어, 자동완성, 오타 교정, 검색 개인화 |
| shop | 가게, 메뉴, 영업시간(`businesshours`), 공지, 정산, 광고, 가게 사진 S3 스토리지, 가게 지갑(엽전 수익) |
| store | 스토어 상품, 주문, 보유 아이템(QR 사용) |
| translation | 채팅/FAQ 번역 — Google Cloud Translation 연동 + DB 캐시 테이블(`translation_cache`) + Redis `@Cacheable` read-through |
| yeopjeon | 엽전 지갑 잔액/거래 이력 — Wallet 이벤트 리스너 보유 |

> **애플리케이션 루트**: `com.chunbaetour.ChunbaeTourApplication`(package선언은 `com.chunbaetour`)이지만, 파일은 물리적으로 `domain/ChunbaeTourApplication.java`에 위치한다(패키지 선언과 디렉터리가 불일치 — Gradle은 컴파일을 허용하지만 관례상 어색한 배치). `com.chunbae.tour`라는 루트 패키지명은 v1.0 설계 당시 가정이었고 실제로는 `com.chunbaetour`다.

### 구조 유지 원칙 (실측 기준)

- `global` 패키지는 없다. 모든 코드는 `domain.*` 아래에 있고, 공통 관심사는 `domain.common`이 담당한다.
- Security/JWT/OAuth는 별도 글로벌 패키지가 아니라 `domain.auth` 내부 하위 패키지(`security`, `jwt`, `oauth`, `config`)다.
- 외부 스토리지(S3/로컬 디스크)는 중앙 인프라 패키지로 묶지 않고, **파일을 다루는 도메인마다 자체 `storage` 하위 패키지**(인터페이스 + `S3*Storage`/`LocalDisk*Storage` 구현체 한 쌍)를 둔다 — `auth.profileimage`, `chat.storage`, `community.free.storage`, `cs.storage`, `shop.storage` 5곳이 동일 패턴을 반복한다.
- 외부 API 클라이언트도 마찬가지로 사용하는 도메인 안에 둔다(`place.client`, `festival.client`, `payment.client`, `translation.client`) — 공용 `global.infra` 패키지는 없다.
- DTO는 Lombok `@Getter` 클래스가 아니라 **Java `record`**로 작성한다(검증 애너테이션은 record 컴포넌트에 직접 붙인다).
- 신규 기능은 먼저 담당 도메인 패키지 안에 배치한다.
- 여러 도메인이 반드시 공유하는 응답/예외/설정만 `common`으로 올린다.
- 도메인 간 참조는 가능하면 ID 기반 논리참조 또는 validator/service를 통해 캡슐화한다.

## 1. 문서 목적

본 문서는 춘배투어 백엔드 애플리케이션의 **실제 코드 기준** 패키지 구조를 정의한다.

프로젝트는 Java 21, Spring Boot, Spring Data JPA, QueryDSL, Redis(Lettuce + Redisson), WebSocket/STOMP, JWT, MySQL 8 기반으로 구현하며, 기능별 책임이 명확하게 분리된 **도메인 중심 패키지 구조**를 따른다. `global`/`domain` 2단 구조였던 v1.0 설계와 달리, 실제 코드는 **모든 것이 `domain` 아래 있고 공통 관심사는 `domain.common`이 흡수**하는 1단 구조로 굳어졌다.

이 문서는 다음을 목표로 한다.

- 팀원별 담당 도메인과 실제 코드 패키지의 일관성 확보
- Controller / Service / Repository / Entity / DTO 책임 분리
- 공통 응답, 공통 예외, 보안, 설정 코드의 위치 통일
- REST API, WebSocket, Redis, 외부 API 연동 코드의 관리 기준 정의
- 추후 기능 확장 시 패키지 구조가 무너지지 않도록 기준 제공

## 2. 패키지 구조 설계 원칙

### 2.1 기본 방향 (실제 코드)

```
com.chunbaetour
├── ChunbaeTourApplication.java   # 패키지 선언은 com.chunbaetour, 파일은 domain/ 아래 위치(불일치)
└── domain
    ├── common      # 공통 응답/예외/설정/레이트리밋/감사로그 — 사실상 global
    └── {21개 도메인 패키지}
```

도메인별로 보통 `controller`, `service`, `repository`, `entity`, `dto`, `type`을 내부에 배치하지만, 도메인 성격에 따라 `event`(도메인 이벤트), `client`(외부 API), `storage`(파일 업로드), `config`(도메인 전용 설정), `constant`를 추가로 둔다. 모든 도메인이 같은 하위 패키지 집합을 갖지는 않는다 — 컨트롤러/서비스가 필요 없는 `banner`(엔티티만), 서브도메인 8개를 묶은 `admin` 같은 예외가 있다.

### 2.2 계층 책임

| 계층 | 역할 | 작성 기준 |
| --- | --- | --- |
| Controller | HTTP 요청/응답 처리 | 비즈니스 로직 작성 금지, Service 호출만 수행. 관리자 전용 컨트롤러는 클래스 레벨 `@PreAuthorize("hasRole('ADMIN')")`로 명시 |
| STOMP Controller | `@MessageMapping` 메시지 수신 처리 | `chat.controller.ChatMessageController`, `cs.controller.SupportMessageController` — CONNECT 단계 인증은 `chat.config.StompChannelInterceptor`가 전담 |
| Service | 핵심 비즈니스 로직 | 트랜잭션, 도메인 규칙, 외부 연동 조합 처리 |
| Repository | DB 접근 | Spring Data JPA + QueryDSL(복잡 조회만 별도 `*QueryRepository`) |
| Entity | DB 테이블 매핑 | 비즈니스 상태 변경 메서드 포함, Setter 지양 |
| DTO | 요청/응답 객체 | **Java `record`**, Entity 직접 노출 금지 |
| Type | Enum 관리 | 상태값 등 enum 분리 |
| common | 공통 관심사 | 보안 제외(보안은 auth 소속), 응답/예외/설정/레이트리밋/감사로그/유틸 |

## 4. `domain.common` 패키지 상세 (구 "global" 역할)

### 4.1 common.config

| 클래스 | 역할 |
| --- | --- |
| `AsyncConfig` | `@Async` 실행기 설정 |
| `CacheConfig` | `@Cacheable` 캐시 매니저(번역/관리자 대시보드/축제/동행 리뷰 점수에서 사용) |
| `ClockConfig` | 테스트 가능한 `Clock` 빈 |
| `CustomFunctionContributor` | QueryDSL/Hibernate에 MySQL 네이티브 함수(`ST_Distance_Sphere`, `match_against` 등) 등록 |
| `JpaAuditingConfig` | `@CreatedDate`/`@LastModifiedDate` 활성화 |
| `PublicDataApiConfig` / `PublicDataApiProperties` | 공공데이터(TourAPI) 공통 연동 설정 |
| `QuerydslConfig` | `JPAQueryFactory` 빈 등록 |
| `RedisConfig` | RedisTemplate, RedissonClient 빈 등록 |
| `RedisJsonSerializerFactory` | Redis 값 직렬화 |
| `S3Config` / `S3Properties` | S3 클라이언트 설정(버킷은 도메인별 storage 구현체가 사용) |
| `SwaggerConfig` | OpenAPI 문서 설정 |
| `WebMvcConfig` | 인터셉터/CORS 등 MVC 설정 |

### 4.2 common.error — [11_공통_에러코드_설계서] 참조

`BusinessException`, `ErrorCode`(enum 상수 210개), `GlobalExceptionHandler` — 단일 출처는 `ErrorCode.java`다.

### 4.3 common.response

`ApiResponse<T>`는 `success` 필드가 없는 **record**다. `code`로 성공/실패를 구분한다. (`success`/Lombok 클래스였던 v1.0 설계는 폐기됨 — [11_공통_에러코드_설계서] 1장 참조)

`CursorPageResponse<T>`로 커서 기반 페이지네이션 응답을 통일한다.

### 4.4 common.entity

```java
@Getter
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {
    @CreatedDate @Column(nullable = false, updatable = false) private LocalDateTime createdAt;
    @LastModifiedDate @Column(nullable = false) private LocalDateTime updatedAt;
}
```

`BaseEntity`는 `createdAt`/`updatedAt`만 가진다(id, soft-delete 필드는 없음). `CommonErrorLog`는 외부 API/AI/번역/업로드 실패 상세를 기록하는 엔티티다.

**Soft Delete는 도메인마다 두 방식이 혼재한다(실측, 단일 규칙 아님):**

| 방식 | 적용 도메인 |
| --- | --- |
| `deletedAt` 타임스탬프 컬럼 | `auth.Account`, `community.comment.Comment`, `community.companion.CompanionPost`, `community.free.FreePost`, `notification.Notification`, `shop.Menu` |
| `status` enum에 `DELETED` 값 포함 | `place.Place`(`PlaceStatus.DELETED`/`SOURCE_DELETED` 2단계), `festival.Festival`, `banner.Banner`, `shop.Shop`(`SUSPENDED`/`CLOSED`로 표현, `DELETED` 값은 없음) |

신규 도메인 작성 시 어느 쪽을 따를지는 기존 같은 성격 도메인(콘텐츠류는 status enum, 사용자 액션류는 deletedAt)을 참고해서 결정한다.

### 4.5 common.ratelimit

`RateLimiter`(인터페이스) + `RedisRateLimiter`(구현) + `RateLimitFilter`. 실제 적용처는 12장 외부 연동 표가 아니라 11장 Redis 표를 참조 — `chat.ChatMessageService`, `cs.SupportMessageService`, `place.DirectionController`, 그리고 `auth`의 가입/로그인 IP 제한(`AUTH_014`)이 이 컴포넌트를 쓴다.

### 4.6 common.audit / common.secret

`SecurityAuditLogger`(인증 실패·제재 등 보안 이벤트 로깅), `SecretValidator`(운영 시크릿 값 형식 검증) — v1.0 설계엔 없던 신규 컴포넌트.

## 5. Domain 패키지 상세 (21개 전수)

### 5.1 admin

8개 하위 도메인(`audit`, `banner`, `certification`, `dashboard`, `market`, `place`, `shop`, `user`)을 묶은 패키지. 자체 엔티티는 거의 없고(`audit`의 `AdminActionLog`만 예외), 대상 도메인의 엔티티/리포지토리를 관리자 권한으로 조작하는 Controller+Service+DTO만 갖는다.

- `audit`: `@LogAdminAction` AOP 애너테이션 + `AdminActionLogAspect`로 관리자 행위를 자동 기록(누가/언제/어떤 대상에 무엇을). `AdminActionContext`로 현재 요청의 관리자 식별.
- `banner`/`certification`/`dashboard`/`market`/`place`/`shop`/`user`: 각각 `controller` + `service` + `dto/{request,response}`만 갖고, 엔티티는 원본 도메인(`banner.Banner`, `shop.ShopCertification`, `market.TraditionalMarket`, `place.Place`, `shop.Shop`, `auth.Account`)을 그대로 참조한다.
- 클래스 레벨 `@PreAuthorize("hasRole('ADMIN')")`로 `/api/v1/admin/**` 접근을 제한한다(SecurityConfig의 URL 패턴 제한과 이중으로 명시).

### 5.2 auth

회원가입/로그인/소셜로그인/토큰 재발급/로그아웃/프로필 이미지/계정 정지를 모두 담당. **Security/JWT/OAuth가 별도 global 패키지가 아니라 이 안에 있다.**

```text
auth/
├── Account.java, AccountRepository.java, AccountStatus.java, Role.java, OauthProvider.java
├── *AuthController.java (Admin/Merchant/User 3종 + AuthTokenController)
```

## 7. DTO 작성 규칙

### 7.1 Request / Response 분리, record 사용

```java
public record CreateChatRoomRequest(
        @NotNull Long postId,
        @NotBlank @Size(max = 50) String title,
        String description,
        @Min(2) @Max(50) int maxMembers
) {}
```

**규칙**

- Request/Response DTO는 **Java `record`**로 작성한다(`@Getter`/`@AllArgsConstructor` Lombok 클래스 아님 — v1.0 설계는 폐기됨).
- 검증 애너테이션(`jakarta.validation.constraints.*`)은 record 컴포넌트에 직접 붙인다.
- Entity를 Controller 응답으로 직접 반환하지 않는다.
- DB 컬럼명은 snake_case, API 필드명은 camelCase를 사용한다.

### 7.2 DTO 네이밍

| 목적 | 네이밍 예시(실제 클래스) |
| --- | --- |
| 생성 요청 | `CreateChatRoomRequest`, `FestivalCreateRequest` |
| 수정 요청 | `PatchUserMeRequest`, `ShopUpdateRequest` |
| 목록 응답 | `ChatRoomMemberResponse`(컬렉션), `AdminBannerListResponse` |
| 상세 응답 | `ChatRoomDetailResponse`, `CompanionDetailResponse` |
| 단건 응답 | `NotificationResponse` |
| 페이징 응답 | `CursorPageResponse<T>`(공통), `TraditionalMarketNearbyPageResponse`(도메인 전용 페이지 응답도 혼재) |

## 8. Entity 작성 규칙

### 8.1 기본 규칙

- Entity는 DB 테이블과 직접 매핑한다.
- Setter는 지양하고, 상태 변경 메서드를 작성한다(예: `Wallet.charge()`, `ChatRoom`의 상태 전이 메서드).
- 상태값은 String이 아니라 Enum으로 관리한다.
- 생성/수정 시간은 `BaseEntity`(createdAt/updatedAt만)를 상속해 관리한다.
- **Soft Delete는 8.2 표(4.4절과 동일)처럼 도메인마다 `deletedAt` 컬럼 또는 `status` enum 중 하나를 선택**한다 — 신규 도메인은 같은 성격의 기존 도메인 사례를 따른다.

### 8.2 예시 (실제 코드 발췌 패턴)

```java
@Entity
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class ChatRoom extends BaseEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    private ChatRoomStatus status;   // OPEN / FULL / CLOSED

    public void close() {
        this.status = ChatRoomStatus.CLOSED;
    }
}
```

| 분산 락(Redisson RLock) | `chat.JoinRequestService`, `companionreview.CompanionService`, `payment.ChargeService`/`QrPayService`, `place.Geocoding/ReverseGeocodingService`, `search.PopularSearchService`, `shop.AdApplicationService`, `store.StorePurchaseService` | Redisson |
| `@Cacheable` 캐시 | `admin.dashboard`, `companionreview`(리뷰 점수), `festival`(축제/캘린더), `translation`(번역 결과) | `common.config.CacheConfig` |
| 번역 결과 read-through | `translation.service.TranslationCacheService` | `@Cacheable`, TTL 24h |

> **위치 기반 탐색은 Redis Geospatial을 쓰지 않는다** — MySQL Spatial 함수로 처리한다(5.14절 참조). ADR-003/기술_모음.md의 "Redis 5대 활용" 서술 중 Geospatial 항목은 현재 코드와 불일치하니 별도 정정이 필요하다.

**규칙(실측)**

- 외부 API Client는 **공통 인프라 패키지가 아니라 사용하는 도메인 안**에 둔다 — `global.infra`류 패키지는 없다.

- 외부 API 실패는 `BusinessException(ErrorCode.EXTERNAL_SERVICE_ERROR)`로 변환한다.
- 실패 상세는 필요한 경우 `common.entity.CommonErrorLog`에 저장한다(`TranslationErrorLogWriter` 등).

## 14. 패키지 작성 시 주의사항

### 14.1 금지 사항

- Controller에서 Repository 직접 호출 금지
- Controller에서 Entity 직접 반환 금지
- 도메인 Service에서 다른 도메인 Repository 직접 남용 금지
- 외부 API 호출 코드를 Controller에 작성 금지
- `ErrorCode` 문자열을 코드 곳곳에 직접 하드코딩 금지(`ErrorCode.XXX` enum 참조만)
- WebSocket 메시지에서 클라이언트가 보낸 `senderId`를 그대로 신뢰 금지(STOMP Principal 기준 인증된 사용자 ID 사용)

### 14.2 권장 사항

- 관리자 전용 엔드포인트는 클래스 레벨 `@PreAuthorize("hasRole('ADMIN')")`로 명시한다(URL 패턴 기반 SecurityConfig 제한과 이중 방어).
- 도메인 간 호출이 필요하면 Service 메서드 또는 도메인 이벤트(`event` 패키지)로 분리한다 — 결제→엽전 지갑 반영처럼 이벤트 리스너로 묶인 사례 참고.
- 공통 응답은 항상 `ApiResponse<T>`(record)를 사용한다.
- 에러는 항상 `BusinessException(ErrorCode)`로 처리한다(결제 도메인만 `PaymentException` 서브클래스 사용).
- 파일 업로드가 필요한 신규 도메인은 기존 5개 도메인의 `storage`(인터페이스 + S3/Local 구현) 패턴을 그대로 따른다.
- Redis Key는 문자열 하드코딩 대신 상수 클래스(`PlaceRedisConstants`, `SearchRedisKeys`, `MerchantHomeCacheKeys` 등)로 관리한다.

## 17. 결론

춘배투어 백엔드는 **`global`/`domain` 2단 구조가 아니라, 21개 도메인 패키지 + 공통 관심사를 흡수한 `common` 패키지로 이뤄진 1단 구조**다. Security/JWT/OAuth는 `auth` 안에, 파일 스토리지와 외부 API 클라이언트는 사용하는 도메인 안에 분산돼 있다 — 이는 의도적 설계 변화이며(중앙 인프라 패키지의 의존성 집중을 피하는 방향), v1.0 설계서가 가정했던 구조와는 다르다.

이 구조를 따르면 팀원별 담당 영역이 명확해지고, 도메인 하나를 통째로 옮기거나 삭제하기 쉽다는 장점이 있다. 다만 S3 storage 패턴처럼 5개 도메인이 똑같은 코드를 반복하는 곳은 공용화 여지가 있다는 점, 그리고 ADR/기술 문서에 적힌 "Redis Geospatial" 같은 서술이 실제 구현과 갈린 부분이 있다는 점은 이번 전수 검증으로 드러난 별도 정리 대상이다.
