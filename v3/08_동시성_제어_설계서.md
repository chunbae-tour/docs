## 1. 목적과 범위

춘배투어는 단일 Spring Boot 애플리케이션이지만 운영 환경에서는 여러 ECS Fargate task가 동시에 요청과 스케줄러를 처리할 수 있다. JVM 내부의 `synchronized`만으로는 인스턴스 간 경쟁을 막을 수 없으므로, 데이터의 중요도와 충돌 형태에 따라 Redis 원자 연산, Redisson 분산 락, DB 비관적 락, JPA 낙관적 락, 조건부 UPDATE(CAS), DB UNIQUE 제약, ShedLock을 조합한다.

본 문서는 구현된 동시성 정책만 기록한다. v2 문서의 예시 코드나 초기 가정이 실제 코드와 다를 경우 v3와 실제 코드를 우선한다.

### 1.1 동시성 설계 목표

| 목표 | 설명 |
| --- | --- |
| 금전·재고 정합성 | 엽전 잔액 음수, 재고 초과 차감, 중복 충전, 이중 환불을 방지한다. |
| 상태 전이 단일성 | PENDING 요청이 동시에 여러 종착 상태로 전이되지 않게 한다. |
| 고빈도 통계 처리 | 조회수·좋아요·검색 순위를 DB 행 락 없이 빠르게 집계한다. |
| 다중 인스턴스 안전성 | ECS task가 여러 개여도 스케줄러와 임계구역이 중복 실행되지 않게 한다. |
| 장애 시 재처리 가능성 | 외부 API·Redis·DB 일부 실패가 데이터 영구 유실로 이어지지 않도록 재시도 단서를 남긴다. |
| 락 범위 최소화 | 외부 API 호출과 사전 검증은 가능한 락 밖에서 수행해 처리량과 장애 격리를 확보한다. |

### 1.2 정합성 수준 분류

| 등급 | 대상 | 요구 수준 | 주요 전략 |
| --- | --- | --- | --- |
| A | 결제, 지갑, 재고, 정산, 환불 | 강한 정합성 | DB 비관적 락, 조건부 UPDATE, UNIQUE, 트랜잭션 |
| B | 채팅 정원, 동행 기간, 관리자 상태 전이 | 충돌 시 한 요청만 성공 | Redisson + DB 락, `@Version`, 상태 가드 |
| C | 관광지 조회수·좋아요 통계 | 빠른 응답 후 최종 수렴 | Redis Lua/Counter + Dirty Set + Write-Behind |
| D | 인기·최근 검색어, 자동완성 | 가용성 우선, 일부 집계 누락 허용 | Redis 원자 연산, fail-open |
| E | 로그인/회원가입 Rate Limit | 보안 정책 우선 | Redis Lua, 장애 시 fail-closed |

## 2. 사용 전략과 선택 기준

| 전략 | 실제 용도 | 장점 | 주의점 |
| --- | --- | --- | --- |
| Redis 원자 연산 | ZSet score 증가, Counter, `SET NX`, List 관리 | 빠르고 단일 명령 경쟁이 없다. | Redis 장애와 영속성 정책을 고려해야 한다. |
| Redis Lua | 최근 검색어, 카운터 seed+증감, lock owner 해제, Rate Limit | 여러 Redis 명령을 하나의 원자 구간으로 묶는다. | Redis Cluster에서는 모든 `KEYS`가 같은 slot이어야 한다. |
| Redisson `RLock` | 상품 구매, 충전 한도, QR 결제, 채팅 참여, 동행 기간, 랭킹 리셋 | 여러 애플리케이션 인스턴스의 임계구역을 직렬화한다. | 락 획득 실패·Redis 장애 정책과 트랜잭션 종료 시점을 명확히 해야 한다. |
| DB 비관적 락 | 상품, 지갑, QR 요청, 결제 주문, 환불, 인증·정산·광고 상태 | 최종 영속 데이터의 stale read와 lost update를 막는다. | 락 순서 불일치 시 데드락 가능성이 있다. |
| JPA `@Version` | `ChatRoom`, `Shop`, `ShopWallet`, `Report`, `Place` | 충돌이 드문 수정에서 last-write-wins를 방지한다. | 충돌 예외를 409로 변환하고 재시도 정책을 정해야 한다. |
| DB 조건부 UPDATE | 결제 상태, 회원 탈퇴, 참여 종료, 신청 취소 | `WHERE 현재상태 = 기대상태`를 DB 단일 명령으로 보장한다. | 영향 행 수 0을 명시적 비즈니스 충돌로 해석해야 한다. |
| DB UNIQUE | 멱등성 키, pending key, 채팅방/게시글, 사용자 찜, 지갑 등 | 애플리케이션 사전검사를 통과한 최종 race를 DB가 차단한다. | 제약명별 예외를 정확한 에러 코드로 변환해야 한다. |
| ShedLock(JDBC) | 통계 동기화, 환불 재시도, 고아 주문 복구, 만료·정리 배치 | 다중 ECS task 중 한 task만 스케줄을 실행한다. | `lockAtMostFor`가 실제 최악 실행 시간보다 길어야 한다. |

### 2.1 공통 원칙

1. 애플리케이션 사전검사는 사용자 경험과 불필요한 I/O 감소를 위한 1차 방어다.
2. 최종 정합성은 DB 락, 조건부 UPDATE, UNIQUE 제약으로 보장한다.
3. Redisson 락이 필요한 경우 트랜잭션 커밋 전에 락을 풀지 않는다.
4. 여러 행을 잠글 때는 정렬된 식별자와 고정된 도메인 순서를 사용한다.
5. DB 커밋과 Redis/S3 같은 외부 부작용의 순서를 맞추기 위해 `TransactionSynchronization.afterCommit/afterCompletion`을 사용한다.

| --- | --- | --- | --- |
| 스토어 구매 | 재고 1개에 동시 구매 | Redis DECR 선점 + 사용자 단위 Redisson + 상품 FOR UPDATE | DB 재고 재검증 |

## 4. 스토어 상품 구매와 지갑

### 4.1 문제

동일 상품에 구매가 몰리면 재고가 음수가 될 수 있고, 동일 사용자의 중복 요청은 지갑을 여러 번 차감할 수 있다. Redis만 기준으로 삼으면 캐시 유실·드리프트 때 실제 재고와 달라질 수 있으므로 DB를 최종 기준으로 유지한다.

### 4.2 실제 3단계 구조

```text
구매 요청
  1. stock:{productId}가 있으면 Redis DECR로 빠른 선점
  2. purchase:lock:{userId} Redisson 락으로 사용자 요청 직렬화
  3. Product SELECT FOR UPDATE로 실제 재고 재검증·차감
  4. Wallet SELECT FOR UPDATE로 잔액 재검증·차감
  5. 주문·보유 아이템 저장
  6. DB 커밋 성공 후 상품 캐시 무효화
```

| 항목 | 구현 |
| --- | --- |
| Redis 재고 키가 없음 | DB 단독 검증으로 진행한다. |
| Redis 차감 후 DB 롤백 | `afterCompletion`에서 Redis 재고를 복구한다. |
| 락 대기 | 3초, leaseTime 미지정으로 Redisson watchdog 사용 |
| DB 락 순서 | Product → Wallet |
| 캐시 무효화 | DB 커밋 성공 후 `product:{productId}` 삭제 |

Redis 연결 자체가 실패하는 경우까지 DB fallback을 보장하는 구조는 아니다. 구현된 fallback은 Redis 재고 키가 없을 때이며, Redis 장애 정책과 혼동하지 않는다.

### 4.3 지갑 정합성

`WalletService`의 충전, 환불 회수, 스토어 구매, 광고 연장은 모두 `findByUserIdWithLock`으로 지갑 행을 잠근 뒤 잔액을 변경한다. 잔액 변경과 `yeopjeon_histories` 저장은 같은 DB 트랜잭션에 포함된다.

| 흐름 | 고정 락 순서 |
| --- | --- |
| 충전 완료 | PaymentOrder → Wallet |
| 환불 승인 | Refund → PaymentOrder → Wallet |
| 스토어 구매 | Product → Wallet |
| 광고 연장 | AdApplication → Wallet |
| QR 결제 승인 | Wallet → ShopWallet |

신규 지갑 생성은 사전 `existsByUserId` 후 저장하되, 동시 생성 race는 `uk_wallets_user_id` UNIQUE 위반을 해당 지갑이 이미 생성된 것으로 해석해 멱등 처리한다.

## 5. 충전·웹훅·QR 결제

### 5.1 충전 요청과 일일 한도

같은 사용자가 서로 다른 Idempotency-Key로 충전을 동시에 요청하면 각각 같은 일일 누적액을 읽고 한도를 초과할 수 있다. `charge:lock:{userId}` Redisson 락으로 일일 한도 SUM 조회부터 PENDING 주문 저장까지 직렬화한다. 락은 트랜잭션 `afterCompletion`에서 해제해 다음 요청이 커밋된 PENDING을 보게 한다.

멱등성은 두 계층으로 보장한다.

| 계층 | 역할 |
| --- | --- |
| Redis `idempotency:{key}` | `SET NX`, 24시간 TTL로 빠른 중복 요청 차단 |
| DB `payment_orders.idempotency_key` UNIQUE | Redis TTL 만료·유실 이후 동일 키 재사용 최종 차단 |

PENDING 주문에 같은 키가 재전송되면 기존 응답을 재생하고, 종착 상태 주문의 같은 키 재사용은 `PAY_007`로 거절한다.

### 5.2 웹훅 멱등성과 DB CAS

PortOne은 at-least-once 방식으로 같은 이벤트를 재전송할 수 있다.

```text
webhook:{webhookId} SET NX
  → PG 실제 상태·금액 검증
  → UPDATE payment_orders
       SET status = COMPLETED
       WHERE order_uid = ? AND status IN (PENDING, FAILED)
  → 영향 행 1일 때만 Wallet 충전
```

`completeIfNotCompleted`, `failIfPending`, `cancelIfPending`은 DB 조건부 UPDATE 결과를 CAS처럼 사용한다. 동시 웹훅과 사용자 취소가 경합해도 한 요청만 영향 행 1을 얻고, 나머지는 중복/상태 충돌로 종료한다. Redis 멱등 키 해제는 DB 커밋 후 실행해 DB 롤백과 Redis 상태가 어긋나지 않게 한다.

### 5.3 QR 결제 요청 생성

1. 가게 행을 FOR UPDATE로 읽어 QR nonce 재발급과 검증을 직렬화한다.
2. 만료된 PENDING을 bulk update로 EXPIRED 처리하고 `pendingKey`를 null로 해제한다.
3. 동일 사용자·가게의 유효 PENDING을 사전 확인한다.
4. `pending_key = userId_shopId` UNIQUE로 최종 race를 차단한다.
5. 요청 시점 잔액은 빠른 실패용으로만 확인하고 실제 차감은 승인 시 재검증한다.

### 5.4 QR 승인·거절·취소

승인·거절·취소는 `qr:lock:{shopId}:{userId}`로 직렬화하고, 락 획득 후 `findByPayRequestIdWithLock`으로 최신 상태를 다시 읽는다. APPROVE 시 Wallet → ShopWallet 순서로 행을 잠그고 금액 이동과 양쪽 이력을 같은 트랜잭션에서 저장한다.

| 현재 상태 | 허용 전이 |
| --- | --- |
| PENDING | COMPLETED, REJECTED, CANCELLED, EXPIRED |
| COMPLETED/REJECTED/CANCELLED/EXPIRED | 추가 전이 거절 |

### 5.5 고아 주문과 환불 복구

- `PendingOrderReconcileScheduler`: 5분 fixedDelay, 생성 후 10분 지난 PENDING을 PG 상태와 대조한다. 30분 이상 미결제면 FAILED로 정리한다.
- `RefundRetryScheduler`: PENDING 환불과 FAILED 환불을 각각 1분 주기로 처리한다. 재시도 간격은 1, 5, 15, 60, 240분이며 최대 5회 후 `REQUIRES_ADMIN`으로 전환한다.
- 스케줄러 자체 중복은 ShedLock으로 막고, 실제 상태 전이는 DB CAS/트랜잭션으로 다시 방어한다.

## 6. 채팅방과 동행

### 6.1 참여 신청과 정원

`JoinRequestService`는 채팅방별 `chatroom:lock:{chatRoomId}`를 사용한다.

| 설정 | 값/정책 |
| --- | --- |
| waitTime | 3초 |
| leaseTime | 5초 |
| transaction timeout | 4초, lease보다 짧게 설정 |
| Redis 장애 | `EXTERNAL_SERVICE_ERROR`로 실패 처리 |

락 밖에서 사용자·방장 같은 사전 조건을 검사하고, 락 안에서 신청/방장 권한을 재검증한다. 신청 승인 시 JoinRequest와 ChatRoom을 각각 FOR UPDATE로 읽고 정원을 증가시킨다. 취소·거절은 `WHERE status=PENDING` 조건부 DELETE/UPDATE를 병행해 승인 경합을 막는다.

`ChatRoom.version`은 방장 위임, 강퇴, 설정 변경 같은 충돌이 드문 수정에서 낙관적 락으로 last-write-wins를 방지한다.

### 6.2 동행 기간 겹침

동행 생성과 참여자 추가는 사용자별 `trip-overlap:user:{userId}` MultiLock을 사용한다. 사용자 ID를 정렬한 뒤 MultiLock을 구성해 서로 다른 요청의 락 획득 순서 차이로 생길 수 있는 데드락 가능성을 줄인다.

```text
참여자 목록 중복 제거·검증
  → userId 정렬
  → 참여자 전원 MultiLock
  → 진행 중 동행 기간 재조회
  → 겹침 없을 때 companion/participant 저장
```

채팅방당 동행 1건과 동행·사용자 조합은 DB UNIQUE 제약이 최종 방어한다. 동행 종료 배치·참여자 추가·동행 취소는 Companion row lock 또는 조건부 UPDATE로 경합을 처리한다.

## 7. 관광지·검색 Redis 동시성

### 7.1 관광지 상세 캐시 Stampede

관광지 상세 캐시 miss 시 `lock:place:{placeId}`를 `SET NX`, TTL 3초, UUID owner token으로 획득한다. 락 보유자만 DB를 조회해 캐시를 채우고, 해제는 Lua로 저장된 owner token이 일치할 때만 삭제한다.

락을 얻지 못한 요청은 50ms 간격으로 최대 20회, 총 1초 동안 캐시를 polling한다. 이후에도 캐시가 없으면 가용성을 위해 DB를 직접 조회한다. 따라서 정상적인 stampede는 줄이지만, 락 보유 작업이 지연되면 제한적으로 DB fallback이 동시 발생할 수 있다.

### 7.2 지오코딩·리버스 지오코딩

동일 주소·좌표에 대한 Kakao API fan-out을 막기 위해 Redisson 락과 double-checked cache lookup을 사용한다.

| 기능 | 캐시 키 | TTL | 락 |
| --- | --- | --- | --- |
| 주소 → 좌표 | `geocoding::{SHA-256(query)}` | 24시간 | 5초 대기, 10초 lease |
| 좌표 → 행정구역 | `region:{lat4}:{lng4}` | 24시간 | 5초 대기, 10초 lease |

캐시 저장은 best-effort다. Redis 장애 시 외부 API 직접 호출로 fallback하므로 핵심 검색 흐름은 유지되지만, 장애 중 동일 요청 fan-out 가능성은 알려진 한계다.

### 7.3 인기 검색어

1. IP를 SHA-256으로 해시한다.
2. `search:log:{hashedIp}:{keyword}`를 `SET NX`, TTL 10분으로 저장한다.
3. 최초 검색만 `ZINCRBY search:ranking 1 keyword`로 집계한다.

ZSet score 증가는 Redis 원자 연산이므로 동시 검색의 lost update를 막는다. 집계 실패는 실제 검색 결과를 막지 않는 fail-open 정책이다.

자정 랭킹 초기화는 `lock:search:ranking:reset` Redisson 락으로 한 인스턴스만 실행하고 `RENAME`으로 오늘 랭킹을 전일 스냅샷으로 원자 교체한다. 새 키 `{search:ranking}:prev`가 만들어지기 전 배포 구간에는 legacy `search:ranking:prev`를 읽기 전용 fallback으로 사용한다.

### 7.4 최근 검색어와 Rate Limit

최근 검색어는 Lua로 `LREM → LPUSH → LTRIM → EXPIRE`를 한 번에 실행해 중복 제거, 최신순 이동, 최대 10개, TTL 30일을 원자적으로 보장한다. Redis 장애는 검색 자체를 막지 않고 저장을 생략한다.

Rate Limit은 Lua로 `INCR → 최초 EXPIRE → PTTL`을 묶는다. 보안 정책이므로 Redis 장애 시 30초 재시도 안내와 함께 fail-closed로 차단한다. 같은 Redis 장애라도 데이터 성격에 따라 fail-open/fail-closed 정책을 다르게 적용한다.

### 7.5 관광지 조회수·좋아요 Write-Behind

조회수·좋아요는 DB에 매 요청 UPDATE하지 않고 Redis counter에 먼저 반영한다.

```text
요청
  → Lua로 counter가 없으면 DB 값 seed 후 INCR/DECR
  → place:dirty_stats Set에 placeId 등록
  → 1분 fixedDelay 배치가 SCAN
  → 100개 chunk로 DB batch update
  → Redis 값이 그대로일 때만 dirty marker 제거
```

| 보호 장치 | 내용 |
| --- | --- |
| counter seed+증감 | Lua 단일 key 원자 연산 |
| 좋아요 음수 방지 | DECR 후 0 미만이면 0으로 보정 |
| DB batch | chunk별 별도 TransactionTemplate, id 정렬 |
| 조회수 역전 방지 | `GREATEST(view_count, redisValue)` |
| 처리 상한 | 45초 또는 10,000건, 나머지는 다음 주기 |
| 스케줄 중복 | ShedLock `place_stats_sync` |
| cleanup 경합 | 제거 전후 counter 재확인, 변경 시 marker 복구 |
| 손상 값 | counter 파싱 실패·누락 시 dirty marker 유지 |

현재 Redis Cluster 호환을 위해 여러 place counter를 MGET하지 않고 단일 GET으로 읽는다. counter 갱신과 전역 dirty set 등록은 서로 다른 key의 별도 명령이므로 완전한 원자 트랜잭션은 아니다. dirty 등록 실패는 ERROR 로그와 메트릭으로 감지해야 하며, 운영상 남은 위험으로 명시한다.

## 8. 관리자·상점·회원 상태 전이

### 8.1 비관적 락 적용

- 상인 신청 승인/거절과 Account role 변경
- 가게 인증 승인/거절/취소와 `isCertified` 변경
- 정산·광고 신청 중복 PENDING 검사와 관리자 처리
- 가게 PROFILE/GALLERY 이미지 순서 계산과 교체
- 신고 제재 대상 Account, 환불/주문 상태

특히 MySQL REPEATABLE READ에서 일반 SELECT가 오래된 snapshot을 볼 수 있는 구간은 FOR UPDATE current read로 최신 커밋 데이터를 확인한다.

### 8.2 낙관적 락 적용

`Shop`, `ShopWallet`, `ChatRoom`, `Report`, `Place`는 `@Version`을 사용한다. 충돌 예외는 공통 예외 처리에서 `COMMON_009(CONCURRENT_UPDATE)` 409로 변환해 조용한 덮어쓰기를 막는다.

### 8.3 조건부 UPDATE와 UNIQUE

- 회원 탈퇴: `deleted_at IS NULL`일 때만 상태를 변경하는 CAS UPDATE
- 동행 참여 종료: `ended_at IS NULL`일 때만 갱신
- 결제/신청 상태: 기대 상태가 PENDING일 때만 UPDATE/DELETE
- 닉네임·이메일·OAuth identity·지갑·pending key: DB UNIQUE가 최종 race 차단

외부 부작용인 토큰 무효화, 감사 로그, 캐시 삭제, S3 삭제는 가능한 DB 커밋 후 실행한다.

## 9. 스케줄러 동시 실행 제어

ShedLock은 JDBC provider를 사용해 여러 ECS task 중 한 task만 동일 스케줄을 실행하게 한다.

| 작업 | 주기 | lockAtMostFor / lockAtLeastFor |
| --- | --- | --- |
| 관광지 통계 동기화 | 이전 실행 종료 후 1분 | 5분 / 10초 |
| PENDING 환불 처리 | 이전 실행 종료 후 1분 | 3분 / 30초 |
| FAILED 환불 재시도 | 이전 실행 종료 후 1분 | 3분 / 30초 |
| 고아 PENDING 충전 복구 | 이전 실행 종료 후 5분 | 10분 / 30초 |
| 사용자 정지 만료 | 매시 정각 | 10분 / 30초 |
| 동행 자동 종료 | 매일 자정 KST | 10분 / 1분 |
| 자유게시글/가게 이미지 orphan cleanup | 이전 실행 종료 후 1시간 | 30분 / 1분 |

Festival/Place/Market 외부 데이터 수집도 분산 실행 가드를 사용한다. ShedLock은 중복 실행을 줄이는 1차 장치이며, 개별 row 상태 전이와 DB 제약은 여전히 멱등하게 설계한다.

## 11. Redis 배치와 slot 정책

현재 `RedisConfig`의 Redisson 연결은 운영 ElastiCache 구성 endpoint를 `useSingleServer()`로 연결하는 단일 샤드 기준이다. 동시에 검색 랭킹과 통계 key는 향후 cluster-mode 또는 cluster 호환 이관 시 CROSSSLOT을 피하도록 일부 hash tag/단일-key 전략을 반영했다.

다음 두 표현을 구분한다.

- 현재 런타임 연결 방식: 단일 서버/단일 샤드 Redisson 설정
- key 설계 호환성: `RENAME` 대상 hash tag, place counter 단일 GET 등 cluster-safe 패턴 일부 적용

운영 구성을 Cluster Mode Enabled로 전환할 경우 `useClusterServers()` 변경, 전체 key slot 검토, SCAN 노드 범위, Lua multi-key 검증이 별도로 필요하다.

## 12. 테스트와 검증 기준

| 테스트 | 기대 결과 |
| --- | --- |
| 재고 1개 동시 구매 | DB 주문은 재고 범위 안에서만 생성되고 Redis는 커밋/롤백 결과와 수렴 |
| 지갑 동시 차감 | 잔액 음수 없음, 이력 balanceSnapshot 일관 |
| 동일 webhook 동시 수신 | Wallet 충전 1회, 주문 COMPLETED 1회 |
| QR 승인·취소 경합 | 하나의 종착 상태만 반영 |
| 채팅 마지막 자리 동시 승인 | 최대 정원 초과 없음 |
| 겹치는 동행 동시 생성 | 동일 사용자는 하나만 성공 |
| 관광지 counter가 배치 중 변경 | dirty marker 유지 후 다음 배치 재처리 |
| Redis counter 손상/누락 | marker를 삭제하지 않고 운영 로그 발생 |
| 여러 인스턴스 스케줄 실행 | ShedLock으로 한 task만 진입 |
| 낙관적 락 충돌 | 409 `COMMON_009` 반환 |

테스트 환경은 Testcontainers MySQL 8.4와 Redis 7을 사용하며, 시간 기반 정책은 운영과 같은 `Asia/Seoul` JVM timezone을 사용한다.

## 13. 알려진 한계와 후속 과제

1. 관광지 counter 변경과 dirty marker 등록은 서로 다른 Redis key 명령이라 완전한 원자성이 아니다. ERROR 로그와 `place.stats.sync.*` 메트릭 기반 운영 감시가 필요하다.
2. 지오코딩 Redis 장애 fallback은 외부 API fan-out을 만들 수 있다. 외부 API별 circuit breaker 또는 로컬 single-flight를 검토할 수 있다.
3. 현재 Redisson은 단일 서버 설정이다. Redis Cluster Mode Enabled 전환 시 client 설정과 전체 key space 검증이 필요하다.
4. 분산 락은 DB 제약을 대체하지 않는다. 새로운 상태 전이에는 반드시 DB 최종 방어를 함께 설계한다.
5. fixed window Rate Limit은 window 경계에서 burst를 허용한다. 요구가 강화되면 token bucket/sliding window로 교체한다.
