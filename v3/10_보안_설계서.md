## 0. 보안 설계 원칙

- **Defense-in-Depth**: 네트워크·인증·인가 계층에 중복 방어.
- **최소 권한**: 보안 그룹·IAM·DB 접근을 필요한 주체에만 허용.
- **무상태 인증**: 토큰 서명 검증 중심 → 수평 확장 대응.
- **시크릿 외부화**: 자격 증명을 코드/이미지에 두지 않음.
- **공격면 최소화**: 불필요한 인바운드(SSH 등) 제거.

## 1. 인증 (Authentication) — ✅ 구현 확인

### 1.1 소셜 로그인 (OAuth 2.0)
- Kakao·Naver OAuth 2.0, 콜백 직접 수신 + 자체 JWT 발급(가입 흐름 직접 제어).
- `state` CSRF 방어, `redirect_uri` 통일·검증.

### 1.2 JWT 이중 토큰 — `application.yml:64-65`, `RefreshTokenStore.java:48-112`, `RefreshCookieFactory:62`
- **Access PT30M / Refresh P7D**.
- **Access**: 응답 body로 전달(무상태, 서명 검증).
- **Refresh**: **HttpOnly + Secure(prod) + SameSite=Lax 쿠키**, `Path=/api/v1/auth`.
- 무효화: **화이트리스트 + Lua 기반 CAS 회전** + `AccessTokenBlacklist` + `LogoutTokenStore`(Lua 원자 연산) → 탈취 토큰 재사용 차단.

### 1.3 이메일 가입 — `PasswordHasher.java:9`, `SignupRequest.java:8-24`
- **BCrypt(strength 10)** 해싱.
- 비밀번호 정책: 최소 8자 + 영문 + 숫자 + 특수문자(정규식 검증).

## 2. 인가 (Authorization) — ✅ 구현 확인 — `SecurityConfig:122-222`, `:70`
- **RBAC 이중 방어**: URL 레벨 `hasRole`(SecurityConfig:122-222) + 메서드 레벨 `@PreAuthorize` **13곳** (`@EnableMethodSecurity`) — 이중 방어 실재.
- 역할: `USER` / `MERCHANT` / `ADMIN`.
- **소유자 검증**: `isOwnedBy()`(예: `FreePostService:72,82`) + `@AuthenticationPrincipal`.
- **ADMIN 행위 감사**: `@LogAdminAction`.
- 응답 코드: `401`(인증 실패) / `403`(권한 부족) 구분.

---

## 3. 전송 구간 보안 (Transport) — ✅ 구현 확인 — `SecurityConfig:101-121`
- **ALB에서 ACM TLS 종료**, 외부는 HTTPS(:443)만 수신, HTTP→HTTPS 리다이렉트.
- **보안 응답 헤더 전부 적용**: HSTS · CSP(`frame-ancestors none`) · X-Frame-Options DENY · X-Content-Type-Options nosniff · Referrer-Policy + `SecurityHeadersIntegrationTest` 검증.

---

## 4. 네트워크 보안 (Network) — ✅ 구현 확인

### 4.1 보안 그룹 체인 (최소 권한)
- `SG-ALB`(80/443) → `SG-TASK`(8080, ALB만) → `SG-RDS`(3306, TASK만) / `SG-ElastiCache`(6379, TASK만). (상세: 07번 §4)

### 4.2 NAT 회피 + 직접 접근 차단
- 태스크는 퍼블릭 서브넷, SG-TASK가 ALB 외 인바운드 전면 차단. 아웃바운드는 IGW 경유(NAT 미사용).

### 4.3 SSH 제거
- Fargate 전환으로 SSH 대상 서버 없음. 배포는 AWS API → **인바운드 22 부재** → 공격면 축소.

### 4.4 CORS — `SecurityConfig:248-261`, `CorsProperties:43-60`, `SecretValidator:134`
- `CorsConfigurationSource` 사용. base=localhost, **prod=환경변수 필수(미설정 시 부팅 실패)**. **wildcard + credentials 동시 사용 금지** 검증.

## 5. 시크릿 / 자격 증명 관리 (Secrets) — 🟡 부분 — `task-def:25-50`
- **AWS Secrets Manager 24키**(태스크 IAM 역할로 로딩).
- RDS = 비밀번호 인증(min12 검증). Redis = **TLS-only, AUTH 없음(의도된 설계)**.
- ⚠️ **시크릿 자동 로테이션 = 미구현**(현재 수동/문서 관리 `secrets-catalog.md`). → §12 백로그.

## 6. 배포 / CI-CD 보안 (Deployment) — ✅ 구현 확인 — `cd.yml:77-81`
- **GitHub Actions OIDC 페더레이션**(정적 액세스 키 0, main 브랜치 한정).
- 배포 IAM **최소 권한**(ecs 한정 + `iam:PassRole` 조건).
- 이미지 태그 = **commit SHA**(latest 아님). 알림에 민감정보 없음.

## 7. 데이터 보안 (Data) — ✅ 구현 확인

### 7.1 입력 검증 · SQLi · XSS — `@Valid`, JPA/QueryDSL
- **SQL Injection 안전**: JPA/QueryDSL 파라미터 바인딩, 문자열 결합 0건.
- 입력 검증: `@Valid` DTO 제약 위주.
- XSS: API-only(JSON 응답), 서버측 sanitize 없음 — 클라이언트(React) 렌더 책임.
- **React는 JSX 기본 출력을 자동 이스케이프**하므로 사용자 콘텐츠(게시글/채팅/리뷰)의 stored XSS가 기본 차단됨 → "클라 책임" 가정 충족.
- 프론트(React, Vercel 호스팅)는 사용자 콘텐츠를 일반 JSX(`{내용}`)로만 출력하며 `dangerouslySetInnerHTML`·마크다운 HTML 렌더를 사용하지 않음 확인 → **stored XSS 차단 완료**.

### 7.2 파일 업로드 · 저장 암호화 — `ShopImageService:90-124`, `S3ShopImageStorage:64`
- 업로드 보안: **magic-byte 검증 + 5MB 제한 + presigned GET(raw 키 미노출) + IDOR 방지(`belongsToShop`)** 전부 구현.
- ✅ **S3 버킷 at-rest 암호화(SSE) + Block Public Access, RDS at-rest 암호화 모두 활성화 확인**(콘솔, 2026-06-22).

## 8. 결제 보안 (Payment) — ✅ 구현 확인 — `WebhookVerifier:27-59`, `CallbackService:99-103`
- **HMAC-SHA256 서명 검증(파싱 전)** + 타임스탬프 5분 replay 방지 + **상수시간 비교**.
- PG **재조회 + 금액·상태 서버측 재검증**.
- 멱등: webhook-id + DB CAS → 중복 충전 방지.

---

## 9. 어뷰징 / 가용성 보호 — ✅ 구현 확인 — `LoginService:104`
- **로그인 5회/분(IP) 제한 + 계정 정지 체크**.
- RateLimiter가 10개 이상 엔드포인트에 적용, **Redis 장애 시 fail-closed**, 전역 토글 제공.

---

## 10. 로깅 / 감사 / 모니터링 — ✅ 구현 확인 — `SecurityAuditLogger`
- 보안 감사 로깅: 로그인/로그아웃/토큰 발급/회전/rate-limit/탈퇴 등.
- **민감정보 마스킹**: IP 마스킹, User-Agent 절단, **토큰·비밀번호·이메일 미기록** + 회귀 테스트.
- 운영 관측: CloudWatch 로그 + ADOT/AMP/Grafana 메트릭(07번 참조).
