## 0. 최종 에러 코드 운용 기준

에러 코드는 `ErrorCode.java`가 단일 출처(source of truth)다. 문서는 코드를 따라가며 갱신한다 — 문서가 먼저 정해지고 코드가 맞추는 구조가 아니다.

| Prefix | 담당 도메인 | 담당자 |
| --- | --- | --- |
| COMMON | 공통 검증, 커서, rate limit, 외부 서비스 연동, 동시성 충돌 | 전체 |
| AUTH | 인증, 회원/마이페이지, OAuth(카카오/네이버), 제재, 프로필 이미지 | 정민교 |
| PLACE | 관광지, 전통시장, 검색, 찜, 관광지 리뷰, 길찾기 | 김인목 |
| FESTIVAL | 축제(수집·조회·관리) | 김인목 |
| BANNER | 배너(Admin) | 정민교 |
| COMMUNITY | 자유게시판 게시글·댓글·이미지 | 박경화 |
| REPORT | 신고 접수/처리, 제재 이력 | 박경화 |
| CHAT | 채팅방, 참여 신청, 동행 채팅, 채팅 파일 | 임하은 |
| NOTIFICATION | 알림 | 임하은 |
| FAQ | 버튼형 FAQ | 임하은 |
| CS | 1:1 상담(고객센터) | 임하은 |
| CR | 동행(Companion) 생애주기 + 동행 리뷰 | 임하은 |
| PAY | 결제, 엽전 충전/환불, QR 결제 | 신현민 |
| STORE | 스토어 상품, 주문, 보유 아이템 | 신현민 |
| MERCHANT | 상인 계정/신청/인증 | 신현민 |
| SHOP | 가게, 메뉴, 정산, 광고, 가게 사진/공지 | 신현민 |

> v1.1(최초 설계) 대비 변경: `USER_*`, `ADMIN_*` prefix는 코드에 존재하지 않는다. USER 관련 에러는 전부 `AUTH_0xx`로, ADMIN(신고/제재 처리) 에러는 전부 `REPORT_0xx`로 흡수 통합됐다. `FESTIVAL`, `BANNER`, `FAQ`, `CS`, `CR`, `REPORT`, `NOTIFICATION` prefix는 v1.1엔 없던 신규 분리다.

### 공통 응답 원칙

- 비즈니스 오류는 `BusinessException(ErrorCode)`로 표현한다. (결제 도메인만 `PaymentException extends BusinessException`으로 한 단계 더 세분화 — 나머지 도메인은 서브클래스 없이 `BusinessException` 그대로 사용)
- validation 오류는 `GlobalExceptionHandler`에서 필드명 기준으로 구체 코드로 매핑하거나 `COMMON_002`(INVALID_REQUEST)로 폴백한다.
- 서버 내부 상세 원인은 사용자 응답에 노출하지 않고 로그(`log.error`/`log.warn`)에만 남긴다.
- 기존 프론트가 사용하는 `code` 문자열은 호환성 검토 없이 변경하지 않는다.

## 1. 에러 응답 공통 포맷

`ApiResponse<T>`는 record로, **`success` 필드가 없다.** `code`로 성공/실패를 구분한다(`"SUCCESS"` 고정값 vs 도메인 에러 코드). `@JsonInclude(NON_NULL)`이라 `data`가 null이면 응답 JSON에서 해당 키가 빠진다.

```json
{
  "code": "AUTH_001",
  "message": "이메일 또는 비밀번호가 올바르지 않습니다."
}
```

### 성공 응답 포맷 (참고)

```json
{
  "code": "SUCCESS",
  "message": "OK",
  "data": { ... }
}
```

### 공통 응답 클래스 (Java, 실제 코드)

```java
@JsonInclude(JsonInclude.Include.NON_NULL)
public record ApiResponse<T>(String code, String message, T data) {

    private static final String SUCCESS_CODE = "SUCCESS";
    private static final String SUCCESS_MESSAGE = "OK";

    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>(SUCCESS_CODE, SUCCESS_MESSAGE, data);
    }

    public static ApiResponse<Void> success() {
        return new ApiResponse<>(SUCCESS_CODE, SUCCESS_MESSAGE, null);
    }

    public static ApiResponse<Void> error(String code, String message) {
        return new ApiResponse<>(code, message, null);
    }
}
```

## 2. 에러코드 네이밍 규칙

```
{도메인 prefix}_{세자리 숫자}

예시)
AUTH_001   ← 인증/회원 도메인, 001번 에러
CHAT_013   ← 채팅 도메인, 013번 에러
PAY_030    ← 결제 도메인, 030번 에러
```

prefix 정의는 [0. 최종 에러 코드 운용 기준] 표를 따른다. `MERCHANT` = 상인 계정/신청/인증, `SHOP` = 가게 실체/메뉴/정산/광고, `STORE` = 사용자가 구매하는 상품/아이템 영역으로 용어를 구분한다.

## 3. HTTP 상태 코드 매핑 기준

| HTTP 상태 코드 | 의미 | 사용 상황 |
| --- | --- | --- |
| `200 OK` / `201 Created` | 성공 | 정상 처리 / 리소스 생성 |
| `400 Bad Request` | 잘못된 요청 | 유효성 검사 실패, 잘못된 입력값, 허용 안 되는 상태 전이 |
| `401 Unauthorized` | 인증 실패 | 토큰 없음/만료/변조, 로그인 실패, 로그아웃된 토큰 |
| `403 Forbidden` | 권한 없음 | 소유자 아님, 미인증 상인, 정지 계정, 도메인 제재 |
| `404 Not Found` | 리소스 없음 | 존재하지 않는 리소스 조회 |
| `409 Conflict` | 충돌 | 중복 데이터, 이미 처리된 요청, 동시 수정 충돌, 정원 초과 |
| `429 Too Many Requests` | 요청 초과 | Rate Limit 초과(로그인 IP 제한, 길찾기/채팅 메시지 버스트 제한) |
| `500 Internal Server Error` | 서버 오류 | 예상치 못한 서버 오류 |
| `502 Bad Gateway` | 외부 인가서버 오류 | 카카오/네이버 OAuth 토큰 교환·사용자 조회 실패 |
| `503 Service Unavailable` | 서비스 불가 | 외부 API(S3/번역/지도/공공데이터) 장애, 분산 락 획득 실패 |

## 5. 글로벌 예외 처리 (GlobalExceptionHandler — 실제 코드)

```java
@Slf4j
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ApiResponse<Void>> handleBusinessException(BusinessException ex) {
        ErrorCode code = ex.getErrorCode();
        return ResponseEntity.status(code.getStatus())
                .body(ApiResponse.error(code.getCode(), code.getMessage()));
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ApiResponse<Void>> handleValidationException(MethodArgumentNotValidException ex) {
        ErrorCode code = resolveFieldErrorCode(ex); // 필드명 기준 매핑, 없으면 INVALID_REQUEST(COMMON_002)
        return ResponseEntity.status(code.getStatus())
                .body(ApiResponse.error(code.getCode(), code.getMessage()));
    }

    @ExceptionHandler(HandlerMethodValidationException.class)
    public ResponseEntity<ApiResponse<Void>> handleHandlerMethodValidation(HandlerMethodValidationException ex) { ... }

    @ExceptionHandler(ConstraintViolationException.class)
    public ResponseEntity<ApiResponse<Void>> handleConstraintViolation(ConstraintViolationException ex) { ... }

    // 낙관적 잠금 충돌 — 동시 요청이 같은 엔티티를 수정한 경우, 클라이언트 재시도로 해결 가능
    @ExceptionHandler(ObjectOptimisticLockingFailureException.class)
    public ResponseEntity<ApiResponse<Void>> handleOptimisticLocking(ObjectOptimisticLockingFailureException ex) {
        return ResponseEntity.status(ErrorCode.CONCURRENT_UPDATE.getStatus())
                .body(ApiResponse.error(ErrorCode.CONCURRENT_UPDATE.getCode(), ErrorCode.CONCURRENT_UPDATE.getMessage()));
    }

    @ExceptionHandler(AuthenticationException.class)
    public ResponseEntity<ApiResponse<Void>> handleAuthentication(AuthenticationException ex) {
        throw ex; // Security 필터 체인의 AuthenticationEntryPoint로 위임
    }

    // JSON 역직렬화 실패(잘못된 enum 값, 타입 불일치 등) — 400 반환
    @ExceptionHandler(HttpMessageNotReadableException.class)
    public ResponseEntity<ApiResponse<Void>> handleHttpMessageNotReadable(HttpMessageNotReadableException ex) { ... }

    @ExceptionHandler({MethodArgumentTypeMismatchException.class, MissingServletRequestParameterException.class})
    public ResponseEntity<ApiResponse<Void>> handleBadRequestExceptions(Exception ex) { ... }

    // MaxUploadSizeExceededException은 서블릿 multipart 파싱 단계에서 발생.
    // 운영 환경에서는 DispatcherServlet 이전(필터 레벨)에서 던져져 이 핸들러에 도달 못하고 500 반환 가능 — 별도 검증 필요.
    @ExceptionHandler(MaxUploadSizeExceededException.class)
    public ResponseEntity<ApiResponse<Void>> handleMaxUploadSize(MaxUploadSizeExceededException ex) { ... }

    @ExceptionHandler(AccessDeniedException.class)
    public ResponseEntity<ApiResponse<Void>> handleAccessDenied(AccessDeniedException ex) {
        throw ex; // Security 필터 체인의 AccessDeniedHandler로 위임
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiResponse<Void>> handleUnknown(Exception ex) {
        log.error("Unhandled exception", ex);
        return ResponseEntity.status(ErrorCode.INTERNAL_SERVER_ERROR.getStatus())
                .body(ApiResponse.error(ErrorCode.INTERNAL_SERVER_ERROR.getCode(), ErrorCode.INTERNAL_SERVER_ERROR.getMessage()));
    }
}
```

> v1.1 대비: `success` boolean이 빠진 `ApiResponse` record 구조에 맞춰 `ApiResponse.error(code, message)` 2-arg 정적 메서드로 변경됨. `AuthenticationException`/`AccessDeniedException`은 이 핸들러가 응답을 만들지 않고 Spring Security의 EntryPoint/AccessDeniedHandler로 재위임(`throw ex`)한다.

## 7. BusinessException 구조 (실제 코드)

```java
@Getter
public class BusinessException extends RuntimeException {
    private final ErrorCode errorCode;

    public BusinessException(ErrorCode errorCode) {
        super(errorCode.getMessage());
        this.errorCode = errorCode;
    }
}

// 결제 도메인만 서브클래스 분리 — 나머지 도메인은 BusinessException을 직접 throw
public class PaymentException extends BusinessException {
    public PaymentException(ErrorCode errorCode) {
        super(errorCode);
    }
}

// 사용 예시 (일반 도메인)
if (wallet.getBalance() < amount) {
    throw new BusinessException(ErrorCode.INSUFFICIENT_BALANCE);
}

// 사용 예시 (결제 도메인, 멱등 처리 등에서 catch 분기 시 구분 목적)
throw new PaymentException(ErrorCode.EXTERNAL_SERVICE_ERROR);
```

> v1.1 문서엔 `AuthException` 등 도메인별 서브클래스가 다수 있는 것처럼 적혀 있었으나, 실제 코드엔 `PaymentException` 단 하나만 존재한다. 나머지 도메인은 전부 `new BusinessException(ErrorCode.XXX)`를 직접 던진다 — 불필요한 서브클래스를 늘리지 않는 방향으로 굳어진 상태.
