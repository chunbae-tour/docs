## 0. 최종 기능 범위 보강

최종 기능 범위는 다음 도메인을 모두 포함한다.

| 도메인 | 핵심 기능 | 주요 기술/설계 |
| --- | --- | --- |
| Auth/User | JWT, OAuth, 프로필, 마이페이지, 탈퇴, 제재 | Spring Security, JWT, Redis blacklist, OAuth Kakao/Naver |
| Place/Map | 관광지 조회, 상세, 지도 마커, 주변 장소, 길찾기, 지오코딩 | QueryDSL, Redis cache/Geo, Kakao Local/Map Link |
| Search | 통합 검색, 장소/축제 검색, 인기 검색어, 최근 검색어, 자동완성, 오타 교정 | QueryDSL, Redis ZSet/List/cache, Redis Cluster hash tag |
| Market/Festival | 전통시장/축제 조회, 상세, 찜, 관리자 수집 | 공공데이터 API, 스케줄링, 관리자 API |
| Like/Review | 장소/축제/시장 찜, 장소 리뷰 | 공통 UserLike, target validator, JPA |
| Community | 자유/동행 게시글, 댓글, 이미지 | cursor paging, S3, 제재 인터셉터 |
| Chat/Companion | 채팅방, 참여 신청, 메시지, 파일, 동행/리뷰 | WebSocket/STOMP, Redis Pub/Sub, S3 |
| CS/FAQ | FAQ, 상담방, 상담 메시지/파일 | STOMP, S3, 관리자 배정/종료 |
| Payment/Yeopjeon | 충전, 환불, QR 결제, 엽전 잔액/이력 | PortOne V2, Webhook signature, idempotency key |
| Shop/Merchant | 상인 신청, 가게/메뉴/공지/이미지/정산/광고/QR | S3, 관리자 심사, cursor paging |
| Store | 상품, 주문, 보유 아이템, 아이템 QR | 재고/주문 상태 관리, QR token |
| Report/Admin | 신고, 제재, 관리자 대시보드, 감사 로그 | 상태 전이, admin audit log |
| Notification/Translation | 알림, 번역 | Google Translation API, 알림 read/delete |

기능 상세는 기존 본문을 유지하되, 위 표를 최종 구현 기준의 상위 목차로 본다.

### F-AUTH-001 회원가입

| 항목 | 내용 |
| --- | --- |
| **기능 ID** | F-AUTH-001 |
| **기능명** | 회원가입 |
| **우선순위** | P0 |
| **MVP** | ✅ |
| **관련 API** | `POST /auth/signup` |

**동작 방식**
1. 이메일, 비밀번호, 닉네임 입력
2. 이메일 중복 검사 (DB 조회)
3. 닉네임 중복 검사 (DB 조회)
4. 비밀번호 암호화 (BCrypt)
5. 사용자 저장 (role: USER)
6. 201 응답 반환

**유효성 검사 규칙**

| 필드 | 규칙 |
| --- | --- |
| email  | 이메일 형식, 최대 255자, 중복 불가 |
| password  | 8자 이상, 영문+숫자+특수문자 포함 |
| nickname | 2~20자, 중복 불가 |

**비즈니스 규칙**
- 상인은 일반 회원가입으로 USER 계정을 생성한 뒤 `/merchants/apply` 상인 등록 신청을 하고, 관리자 승인 후 role이 MERCHANT로 변경되는 방식으로 처리
- 가입 시 엽전 지갑(Wallet) 자동 생성 (잔액 0)

### F-AUTH-002 소셜 로그인 (OAuth)

| 항목 | 내용 |
| --- | --- |
| **기능 ID** | F-AUTH-002 |
| **기능명** | 소셜 로그인 (카카오·네이버 OAuth 2.0) |
| **우선순위** | P0 |
| **MVP** | ✅ |
| **관련 API** | `GET /auth/oauth/{provider}`, `GET /auth/oauth/{provider}/callback` |

**동작 방식**
1. 카카오·네이버 OAuth 2.0 Authorization Code Flow
2. 소셜 계정 정보 수신 → 이메일 기준 기존 회원 조회
3. 신규: USER 계정 자동 생성 + Wallet 생성
4. JWT Access/Refresh 발급 후 프론트엔드로 리다이렉트

**비즈니스 규칙**
- 소셜 계정 email이 일반 계정과 동일하면 별도 계정으로 분리
- Refresh Token은 Redis에 저장 (TTL 7일)

### F-AUTH-003 로그인 / 토큰 발급

| 항목 | 내용 |
| --- | --- |
| **기능 ID** | F-AUTH-003 |
| **기능명** | 로그인 |
| **우선순위** | P0 |
| **MVP** | ✅ |
| **관련 API** | `POST /auth/login` |

**동작 방식**
1. 이메일로 사용자 조회
2. 계정 정지 여부 확인
3. 비밀번호 BCrypt 비교
4. Access Token (30분) 생성
5. Refresh Token (7일) 생성 → Redis 저장 (key: `refresh:{userId}`)
6. Access Token → Response Body
7. Refresh Token → HttpOnly Cookie

**토큰 구조**
| 토큰 | 유효기간 | 저장 위치 | Claim |
|——|———|———|——-|
| Access Token | 30분 | 클라이언트 메모리 | userId, role, email |
| Refresh Token | 7일 | HttpOnly Cookie + Redis | userId |

**동작 방식**

## 2. 관광지 / 지도 (담당: 김인목)

### F-PLACE-001 위치 기반 탐색

| 항목 | 내용 |
| --- | --- |
| **기능 ID** | F-PLACE-001 |
| **기능명** | 위치 기반 관광지·전통시장 탐색 |
| **우선순위** | P0 |
| **MVP** | ✅ |
| **관련 API** | `GET /places/nearby` |

**동작 방식**
1. 클라이언트에서 현재 위치 (lat, lng) 전달
2. Redis Geospatial에서 반경 내 장소 우선 조회 (캐시 Hit)
3. 캐시 Miss 시 → DB 조회 (Haversine 공식 또는 QueryDSL 반경 쿼리)
4. DB에도 없을 경우 → 외부 TourAPI Fallback
5. 결과 정렬: 거리순 (직선거리)
6. 로그인 사용자: 찜 여부(isLiked) 포함 반환

**Redis Geospatial 구조**

```
GEOADD geo:places {lng} {lat} {placeId}
GEORADIUS geo:places {lng} {lat} {radius} m ASC
```

**성능 전략**
- 동일 좌표·반경 결과는 Redis에 5분 캐싱
- 위치 좌표는 소수점 3자리로 반올림 후 캐시 키 생성 (불필요한 캐시 증가 방지)

## 3. 검색 (담당: 김인목)

### F-SEARCH-001 키워드 검색

| 항목 | 내용 |
| --- | --- |
| **기능 ID** | F-SEARCH-001 |
| **기능명** | 통합 키워드 검색 |
| **우선순위** | P0 |
| **MVP** | ✅ |
| **관련 API** | `GET /search/places`, `POST /search` |

**동작 방식**
1. 키워드 수신
2. DB 검색 (QueryDSL - 관광지명 LIKE 검색)
3. DB 결과 부족 시 → 
3-1. QueryDSL 기반 DB 검색 수행
3-2. 인기 검색어 Redis ZSET 사용
3-3. 외부 검색 API 연동은 추후 확장
4. 검색어 Redis ZSet 집계: `ZINCRBY search:ranking 1 {keyword}`
5. 사용자 최근 검색어 저장: Redis LIST `search:recent:{userId}` (최대 10개)
6. 결과 반환

**인기 검색어 집계 구조**

```
Redis ZSet key: search:ranking
- ZINCRBY: 검색할 때마다 score +1
- ZREVRANGE: TOP 10 조회
- 매일 자정 score 초기화 (일간 랭킹)
```

### F-CHAT-002 실시간 메시지 송수신

| 항목 | 내용 |
| --- | --- |
| **기능 ID** | F-CHAT-002 |
| **기능명** | 실시간 메시지 송수신 |
| **우선순위** | P0 |
| **MVP** | ✅ |
| **관련 API** | WebSocket `/pub/chat/rooms/{chatRoomId}/messages`, `POST /chat-rooms/{chatRoomId}/files` |

**동작 방식**
1. 클라이언트 WebSocket 연결 (`/ws-stomp`)
2. JWT 토큰으로 WebSocket 핸드셰이크 인증
3. 채팅방 구독 (`/sub/chat/rooms/{chatRoomId}`)
4. 메시지 발행 → STOMP 브로커 수신
5. Redis Pub/Sub으로 모든 서버에 메시지 발행
6. 구독 중인 클라이언트에 Push
7. MySQL에 메시지 저장
8. 단, 영구 보관을 전제로 하지 않고 MVP에서는 최근 메시지 조회 중심으로 관리

**메시지 타입**
| 타입 | 설명 |
| — | — |
| TEXT | 일반 텍스트 메시지 |
| IMAGE | 이미지 메시지, S3 URL 기반 전송 |
| FILE | 파일 메시지, S3 URL 기반 전송 |
| SYSTEM | 입장/퇴장/강퇴 알림 |

**파일/이미지 메시지 처리**

- 파일/이미지 메시지는 `POST /chat-rooms/{chatRoomId}/files`로 S3에 먼저 업로드한다.
- 업로드 API는 채팅방 참여자 권한을 검증한 뒤 S3에 저장하고 `fileUrl`, `fileName`, `fileSize`, `contentType`을 반환한다.
- 이후 클라이언트는 WebSocket 메시지 발행 시 `messageType`을 `IMAGE` 또는 `FILE`로 지정하고, 반환받은 `fileUrl`을 함께 전송한다.
- DB에는 S3 URL(`file_url`)과 원본 파일명(`file_name`), 파일 크기(`file_size`)를 함께 저장한다.
- 일반 텍스트 메시지는 `content`를 사용하고, 파일/이미지 메시지는 `content`를 선택 설명 또는 캡션으로 사용할 수 있다.
