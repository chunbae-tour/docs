## ADR-001 Java 21 + Spring Boot 4.0.6 선택

> **상태**: 결정됨
**날짜**: 2026-05-12
**결정자**: 전체 팀
> 

### 컨텍스트

백엔드 언어와 프레임워크를 선택해야 한다. 팀원 전원이 Java / Spring Boot 교육 과정을 이수했으며, 1달이라는 짧은 개발 기간 안에 안정적으로 개발해야 한다.

### 결정

**Java 21 LTS + Spring Boot 4.0.6** 를 선택한다.

### 근거

- 팀원 전원이 Java / Spring Boot에 가장 익숙하여 **러닝 커브 최소화**
- Java 21은 현재 LTS 버전으로 **장기 지원 보장**
- Virtual Thread (Project Loom) 지원으로 **I/O 집약적 작업 성능 개선** (채팅, 외부 API 호출)
- Spring Boot 4.0.6는 Spring Framework 7.x 기반으로 **Jakarta EE 11 완전 지원**

### Trade-off

| 항목 | Java 21 + Spring Boot 4.0.6 | Kotlin + Spring Boot | Node.js (NestJS) |
| --- | --- | --- | --- |
| 팀 친숙도 | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐ |
| 생산성 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 성능 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 생태계 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 1달 개발 적합성 | ✅ | ❌ (Kotlin 학습 필요) | ❌ (전환 비용) |

### 결과

- Spring Security, Spring Data JPA, QueryDSL 등 Spring 생태계를 그대로 활용 가능
- Virtual Thread 활용 시 WebSocket 연결 처리 성능 향상 기대
- ⚠️ Spring Boot 4.0.6는 Spring Framework 7.x 기반으로 Jakarta EE 11을 요구하므로, 기존 `javax.*` 패키지 대신 `jakarta.*` 패키지를 사용해야 함. 라이브러리 의존성 호환성 사전 확인 필요

## ADR-002 MySQL 선택 (관계형 DB)

> **상태**: 결정됨
**날짜**: 2026-05-12
**결정자**: 전체 팀
> 

### 컨텍스트

서비스의 핵심 데이터(사용자, 엽전, 결제, 관광지, 채팅 등)를 저장할 데이터베이스를 선택해야 한다. 데이터 간 관계가 복잡하고 결제 데이터는 높은 정합성이 요구된다.

### 결정

**MySQL 8.4** 를 메인 데이터베이스로 선택한다.

### 근거

- **트랜잭션 지원**: 엽전 결제, 재고 차감 등 정합성이 중요한 금전 처리에 ACID 보장
- **관계 표현**: 사용자-엽전-결제-상품 간 복잡한 관계를 FK로 표현 가능
- **팀 친숙도**: 교육 과정에서 MySQL을 주로 사용
- **AWS RDS 완전 관리형**: 운영 부담 최소화

### Trade-off

| 항목 | MySQL | PostgreSQL | MongoDB |
| --- | --- | --- | --- |
| 팀 친숙도 | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐ |
| 트랜잭션 | ✅ | ✅ | ⚠️ (제한적) |
| 정합성 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| 유연한 스키마 | ❌ | ❌ | ✅ |
| AWS RDS 지원 | ✅ | ✅ | ❌ (Atlas 별도) |
| 1달 적합성 | ✅ | ⚠️ (학습 비용) | ❌ |

### 결과

- 엽전 결제, 재고 차감 등 금전 처리에서 트랜잭션 정합성 보장
- Spring Data JPA + QueryDSL과 자연스럽게 연동
- ⚠️ 채팅 메시지 대용량 저장은 MySQL 부하를 줄이기 위해 별도 전략 검토 필요 (현재는 MySQL 저장)

## ADR-003 Redis 다목적 활용 전략

> **상태**: 결정됨
**날짜**: 2026-05-12
**결정자**: 전체 팀
> 

### 컨텍스트

Redis를 단순 캐시가 아닌 여러 목적으로 활용해야 한다. 각 기능별로 Redis의 어떤 자료구조와 기능을 사용할지 명확히 정의해야 한다.

### 결정

Redis를 **5가지 목적**으로 구분하여 활용한다.

### 활용 목적별 설계

| 목적 | 자료구조 | 키 패턴 | TTL | 담당 도메인 |
| --- | --- | --- | --- | --- |
| 인기 검색어 순위 | ZSet (ZINCRBY) | `search:ranking` | 없음 (주기 초기화) | 검색 |
| 위치 기반 탐색 | Geospatial (GEOADD) | `geo:places` | 없음 | 관광지/지도 |
| 재고 가점유 | String (DECR/INCR) | `stock:{productId}` | 10분 | 스토어 |
| 분산 락 | String (SETNX) | `{domain}:lock:{id}` | 5초 | 결제/채팅 |
| 채팅 메시지 브로커 | Pub/Sub | `chat:{roomId}` | - | 채팅 |
| 고객센터 상담 메시지 브로커 | Pub/Sub | `support:{supportRoomId}` | - | AI FAQ/고객센터 |

### 근거

- 단일 Redis 인스턴스(ElastiCache)로 여러 기능을 처리하여 **인프라 비용 절감**
- 각 기능에 최적화된 Redis 자료구조 활용으로 **성능 최적화**
- DB 접근 최소화로 **MySQL 부하 감소**

### Trade-off

| 항목 | Redis 다목적 활용 | 기능별 별도 인프라 |
| --- | --- | --- |
| 비용 | 낮음 (단일 인스턴스) | 높음 |
| 운영 복잡도 | 낮음 | 높음 |
| 장애 영향 범위 | 넓음 (단일 장애점) | 좁음 |
| 1달 프로젝트 적합성 | ✅ | ❌ |

### 결과

- Redis 장애 시 검색 순위, 위치 기반 탐색, 채팅 등 다수 기능에 영향
- ⚠️ 이를 완화하기 위해 AWS ElastiCache Multi-AZ 구성 적용 (ADR-009 참조)

## ADR-004 JWT 기반 인증 방식 선택

> **상태**: 결정됨
**날짜**: 2026-05-12
**결정자**: 정민교
> 

### 컨텍스트

사용자 인증 방식을 선택해야 한다. PWA + REST API 구조에서 서버 세션 없이 인증을 처리해야 한다.

### 결정

**JWT (Access Token + Refresh Token)** 방식을 선택한다.

### 토큰 전략

| 토큰 | 유효기간 | 저장 위치 | 용도 |
| --- | --- | --- | --- |
| Access Token | 30분 | 클라이언트 메모리 | API 인증 |
| Refresh Token | 7일 | HttpOnly Cookie | Access Token 재발급 |

### 근거

- **Stateless**: 서버에 세션 저장소 불필요 → 수평 확장 용이
- **PWA 친화적**: REST API와 자연스럽게 연동
- **Refresh Token HttpOnly Cookie**: XSS 공격으로부터 Refresh Token 보호

### Trade-off

| 항목 | JWT | 세션 기반 | OAuth2 (소셜 로그인) |
| --- | --- | --- | --- |
| 구현 복잡도 | 중간 | 낮음 | 높음 |
| 확장성 | ✅ (Stateless) | ❌ (세션 공유 필요) | ✅ |
| 보안 | ⚠️ (토큰 탈취 위험) | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 팀 친숙도 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |
| 1달 적합성 | ✅ | ✅ | ❌ |

### 결과

- Spring Security Filter Chain에서 JWT 검증 처리
- Access Token 만료 시 Refresh Token으로 자동 재발급
- ⚠️ Access Token 강제 만료(로그아웃, 강제 탈퇴)는 Redis 블랙리스트로 처리

## ADR-005 WebSocket + STOMP 실시간 채팅

> **상태**: 결정됨
**날짜**: 2026-05-13
**결정자**: 임하은
> 

### 컨텍스트

단체 채팅방의 실시간 메시지 송수신 기능을 구현해야 한다. 여러 사용자가 동시에 메시지를 주고받는 양방향 통신이 필요하다.

### 결정

**WebSocket + STOMP 프로토콜** 을 선택한다.

### 근거

- **WebSocket**: HTTP 폴링 대비 낮은 지연시간, 지속 연결로 실시간성 보장
- **STOMP**: WebSocket 위의 메시징 프로토콜로 `/pub`, `/sub` 구조로 채팅방 구독 관리 용이
- **Spring 공식 지원**: `spring-boot-starter-websocket`으로 간편 통합

### Trade-off

| 항목 | WebSocket + STOMP | SSE | HTTP Long Polling |
| --- | --- | --- | --- |
| 양방향 통신 | ✅ | ❌ (단방향) | ⚠️ (유사 구현) |
| 실시간성 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 구현 복잡도 | 중간 | 낮음 | 낮음 |
| Spring 지원 | ✅ 공식 | ✅ 공식 | ✅ |
| 채팅 적합성 | ✅ | ❌ | ❌ |

### 결과

- WebSocket Endpoint: `/ws-stomp`
- STOMP `/pub/chat/rooms/{chatRoomId}/messages` 로 메시지 발행
- STOMP `/sub/chat/rooms/{chatRoomId}` 로 채팅방 구독
- 고객센터 상담 채팅도 MVP에 포함하며 `/pub/support/rooms/{supportRoomId}/messages`, `/sub/support/rooms/{supportRoomId}` 경로를 사용
- 다중 서버 환경에서 메시지 동기화는 Redis Pub/Sub으로 처리 (ADR-006 참조)

## ADR-006 Redis Pub/Sub 기반 채팅 메시지 브로커

> **상태**: 결정됨
**날짜**: 2026-05-13
**결정자**: 임하은
> 

### 컨텍스트

서버가 여러 대로 확장될 경우, 사용자 A가 서버 1에 연결되고 사용자 B가 서버 2에 연결된 상황에서 같은 채팅방의 메시지가 전달되지 않는 문제가 발생한다.

### 결정

**Redis Pub/Sub** 을 채팅 메시지 브로커로 사용한다.

### 문제 상황

```
[다중 서버 문제]

User A ──► Server 1 (WebSocket 연결)
User B ──► Server 2 (WebSocket 연결)

User A가 메시지 전송 → Server 1만 수신
→ Server 2에 연결된 User B는 메시지를 받지 못함!
```

### 해결 구조

```
User A ──► Server 1 ──► Redis Publish (chat:{roomId})
                              │
                    ┌─────────┴─────────┐
                    ▼                   ▼
                 Server 1           Server 2
              (Subscribe)        (Subscribe)
                    │                   │
                 User A              User B
              (메시지 수신)         (메시지 수신)
```

### Trade-off

| 항목 | Redis Pub/Sub | Kafka | RabbitMQ |
| --- | --- | --- | --- |
| 설정 복잡도 | 낮음 | 높음 | 중간 |
| 메시지 영속성 | ❌ | ✅ | ✅ |
| 성능 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 1달 프로젝트 적합성 | ✅ | ❌ (운영 부담) | ❌ |
| 이미 사용 중인 인프라 | ✅ (Redis 활용) | ❌ | ❌ |

### 결과

- 추가 인프라 없이 기존 Redis(ElastiCache)를 재활용하여 비용 절감
- ⚠️ Redis Pub/Sub은 메시지 영속성이 없어 Redis 장애 시 메시지 유실 가능
- 채팅 메시지는 별도로 MySQL에 저장하여 유실에 대비하되, 영구 보관을 전제로 하지 않고 MVP에서는 최근 메시지 조회 중심으로 관리
- 고객센터 실시간 상담 메시지도 Redis Pub/Sub 전달 + MySQL 저장 구조를 사용한다

## ADR-007 QueryDSL 도입

> **상태**: 결정됨
**날짜**: 2026-05-13
**결정자**: 전체 팀
> 

### 컨텍스트

관광지 탐색, 검색, 커뮤니티 등 다양한 조건의 동적 쿼리가 필요하다. JPA의 JPQL만으로는 복잡한 동적 조건 처리가 어렵다.

### 결정

**QueryDSL** 을 동적 쿼리 생성 라이브러리로 도입한다.

### 근거

- **타입 안전성**: 컴파일 타임에 쿼리 오류 감지 (런타임 오류 방지)
- **동적 쿼리**: `BooleanBuilder`로 조건별 쿼리 유연하게 조합
- **가독성**: JPQL 문자열보다 코드 가독성 우수
- **Spring Data JPA 통합**: `QuerydslPredicateExecutor`로 자연스럽게 연동

### Trade-off

| 항목 | QueryDSL | JPQL | MyBatis | Spring Data Specification |
| --- | --- | --- | --- | --- |
| 타입 안전성 | ✅ | ❌ | ❌ | ✅ |
| 동적 쿼리 | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 가독성 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ |
| 초기 설정 | 중간 | 없음 | 중간 | 없음 |
| 팀 친숙도 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐ |

### 결과

- 관광지 필터 검색, 커뮤니티 게시글 검색 등 다중 조건 쿼리에 활용
- ⚠️ Q클래스 생성을 위한 빌드 설정(APT) 초기 세팅 필요 → 정민교(배포 담당)가 초기 설정 책임

## ADR-008 분산 락 구현체로 Redisson 선택

> **상태**: 결정됨
**날짜**: 2026-05-13
**결정자**: 신현민
> 

### 컨텍스트

엽전 결제, 재고 차감, 채팅방 참여 등 동시성 제어가 필요한 시나리오에서 분산 락을 구현해야 한다. Redis 기반 분산 락 라이브러리를 선택해야 한다.

### 결정

**Redisson** 을 분산 락 구현체로 선택한다.

### 근거

- **RLock 추상화**: `tryLock(waitTime, leaseTime)`으로 직관적인 락 제어
- **자동 갱신(Watch Dog)**: 락 보유 중 leaseTime이 만료되기 전 자동 연장
- **Lua 스크립트 기반**: 락 획득/해제가 원자적으로 처리됨
- **Spring 통합**: `spring-boot-starter-data-redis`와 함께 사용 가능

### Trade-off

| 항목 | Redisson | Lettuce (직접 구현) | Spring Integration |
| --- | --- | --- | --- |
| 구현 편의성 | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ |
| Watch Dog (자동 연장) | ✅ | ❌ (직접 구현) | ❌ |
| 원자적 처리 | ✅ (Lua) | ⚠️ (직접 구현) | ✅ |
| 의존성 추가 | 필요 | 불필요 | 필요 |

### 결과

- `DistributedLockExecutor` 공통 유틸로 추상화하여 각 도메인에서 재사용 (동시성 제어 설계서 참조)
- ⚠️ Redisson은 기본적으로 단일 Redis 노드 기준. ElastiCache Cluster 모드 사용 시 RedissonClusterClient 설정 필요

## ADR-009 AWS 인프라 구성

> **상태**: 결정됨
**날짜**: 2026-05-12
**결정자**: 정민교
> 

### 컨텍스트

서비스를 외부에서 접근 가능하도록 배포하고, 데이터베이스와 캐시를 안정적으로 운영할 인프라를 구성해야 한다.

### 결정

**AWS EC2 + RDS(MySQL) + ElastiCache(Redis) + ECR** 조합을 선택한다.

### 구성 요소별 선택 근거

| 서비스 | 선택 | 근거 |
| --- | --- | --- |
| **EC2** | t3.medium | 애플리케이션 서버. Docker 컨테이너 실행 |
| **RDS MySQL** | db.t3.micro | 완전 관리형 MySQL. 자동 백업, 패치 관리 |
| **ElastiCache Redis** | cache.t3.micro | 완전 관리형 Redis. 클러스터 모드 Off, Multi-AZ On |
| **ECR** | - | Docker 이미지 레지스트리. GitHub Actions CI/CD 연동 |

### Trade-off

| 항목 | AWS | GCP | 자체 서버 |
| --- | --- | --- | --- |
| 팀 친숙도 | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐ |
| 관리형 서비스 | ✅ (RDS, ElastiCache) | ✅ | ❌ |
| 비용 | 중간 | 중간 | 낮음 |
| 교육 계정 지원 | ✅ | ⚠️ | - |

### 결과

- Docker 이미지를 ECR에 Push → EC2에서 Pull & 실행하는 배포 파이프라인 구성
- RDS와 ElastiCache는 VPC 내부에서만 접근 가능하도록 Security Group 설정
- ⚠️ 비용 관리: 개발 기간 중 미사용 시 EC2 인스턴스 중지 권장

## ADR-010 GitHub Actions 기반 CI/CD

> **상태**: 결정됨
**날짜**: 2026-05-12
**결정자**: 정민교
> 

### 컨텍스트

5명이 동시에 개발하는 환경에서 코드 통합과 배포를 자동화해야 한다. 별도 CI/CD 서버 없이 간편하게 구성해야 한다.

### 결정

**GitHub Actions** 로 CI/CD 파이프라인을 구성한다.

### 파이프라인 설계

```
[Push / PR to main]
        │
        ▼
  [CI] 빌드 + 테스트
  (./gradlew build)
        │
        ▼
  [Docker 이미지 빌드]
        │
        ▼
  [ECR Push]
        │
        ▼
  [EC2 SSH 접속]
        │
        ▼
  [ECR Pull + Docker 재시작]
```

### Trade-off

| 항목 | GitHub Actions | Jenkins | GitLab CI |
| --- | --- | --- | --- |
| 별도 서버 필요 | ❌ | ✅ | ✅ (GitLab 사용 시) |
| GitHub 연동 | ✅ 네이티브 | ⚠️ | ❌ |
| 무료 사용 | ✅ (public repo) | ❌ | ⚠️ |
| 설정 복잡도 | 낮음 | 높음 | 중간 |
| 팀 친숙도 | ⭐⭐⭐ | ⭐⭐ | ⭐⭐ |

### 결과

- `main` 브랜치 Push 시 자동 빌드·배포
- PR 단계에서 빌드 실패 시 Merge 차단으로 코드 품질 보호
- ⚠️ GitHub Actions 무료 플랜은 월 2,000분 제한. 빌드 시간 최적화 필요 (Gradle 캐시 활용)

## ADR-011 엽전 시스템 도입 (가상 화폐)

> **상태**: 결정됨
**날짜**: 2026-05-12
**결정자**: 신현민, 전체 팀
> 

### 컨텍스트

외국인 관광객이 전통시장에서 결제할 때 언어 장벽과 현금 문제를 해결해야 한다. 기존 카드 결제는 상인의 수수료 부담이 크고, 현금은 외국인이 준비하기 어렵다.

### 결정

**엽전(가상 화폐)** 시스템을 도입하여 앱 내 전용 결제 수단으로 활용한다.

### 설계 구조

```
[USER]                    [MERCHANT]
  │                           │
  │  실제 결제(PG사)            │
  │──────────────────────►    │
  │  엽전 충전                  │
  │                           │
  │  QR 스캔 + 엽전 결제        │
  │──────────────────────►    │
  │                      엽전 수신
  │                           │
  │                      정산 신청 → 실제 현금 출금
```

### 근거

- **언어 장벽 해소**: QR 스캔 + 엽전 차감으로 언어 없이 결제 가능
- **수수료 절감**: PG 수수료는 충전 시 1회만 발생, 엽전 이체는 수수료 없음
- **바가지 방지**: 표준 가격으로 등록된 메뉴 기반 결제로 임의 가격 변경 불가
- **한국 문화 반영**: “엽전”이라는 네이밍으로 전통 콘셉트와 일치

### Trade-off

| 항목 | 엽전 시스템 | 카드 직접 결제 | 포인트 시스템 |
| --- | --- | --- | --- |
| 언어 장벽 해소 | ✅ | ❌ | ✅ |
| 수수료 절감 | ✅ | ❌ | ✅ |
| 구현 복잡도 | 높음 | 낮음 | 중간 |
| 규제 리스크 | ⚠️ (전자금융업 검토) | 없음 | 낮음 |
| 서비스 차별성 | ⭐⭐⭐⭐⭐ | ⭐ | ⭐⭐ |

### 결과

- 엽전 잔액은 MySQL Wallet 테이블에서 관리, 동시성은 분산 락 + 비관적 락으로 보호 (ADR-008, 동시성 제어 설계서 참조)
- ⚠️ 실제 서비스화 시 전자금융거래법 검토 필요 (현재는 프로젝트 시연 목적)

## ADR-012 PWA 기반 프론트엔드

> **상태**: 결정됨
**날짜**: 2026-05-12
**결정자**: 전체 팀
> 

### 컨텍스트

프론트엔드를 React로 개발하되, 외국인 관광객도 앱 스토어 없이 바로 접근 가능해야 한다. iOS/Android 네이티브 앱 개발 인력이 없다.

### 결정

**React + PWA** 로 프론트엔드를 구성한다. (바이브코딩으로 처리)

### 근거

- **앱 스토어 불필요**: URL 접속만으로 앱처럼 사용 가능 → 외국인 관광객의 설치 장벽 제거
- **홈 화면 추가**: 스마트폰 홈 화면에 아이콘 추가로 네이티브 앱과 유사한 UX
- **GPS 접근**: PWA에서도 브라우저 Geolocation API로 위치 정보 접근 가능
- **React 단일 코드베이스**: iOS/Android 별도 개발 불필요

### Trade-off

| 항목 | React PWA | React Native | Flutter |
| --- | --- | --- | --- |
| 설치 장벽 | ✅ 없음 | ⚠️ 앱스토어 | ⚠️ 앱스토어 |
| 개발 속도 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| 네이티브 기능 접근 | ⚠️ 제한적 | ✅ | ✅ |
| 팀 역량 | ✅ (바이브코딩) | ❌ | ❌ |
| 푸시 알림 | ⚠️ (Web Push) | ✅ | ✅ |

### 결과

- 위치 기반 탐색(Geolocation API), 카메라(QR 스캔) 등 핵심 기능은 브라우저 API로 구현 가능
- ⚠️ iOS Safari의 PWA 지원이 Android Chrome 대비 제한적 → 주요 기능 iOS Safari 호환성 테스트 필요
- 알림은 Web Push API 기반으로 구현 (브라우저 지원 범위 내)
