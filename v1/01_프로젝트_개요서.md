## 2. 프로젝트 주제

저희 프로젝트는 **위치기반 관광지 매칭·투어 플랫폼**입니다.

이 서비스는 사용자의 현재 위치를 기반으로 주변 관광지와 전통시장 정보를 제공하고, 길찾기, 축제 정보, 커뮤니티, 채팅, 매칭 등의 기능을 통해 더 편리하고 즐거운 여행 경험을 제공하는 것을 목표로 합니다.

---

## 3. 프로젝트 기획 배경

외국인 관광객이 한국을 여행할 때는 여러 불편함을 겪을 수 있습니다.

어떤 관광지를 가야 할지 찾기 어렵거나, 전통시장에 대한 정보가 부족하거나, 이동 경로를 확인하기 번거로울 수 있습니다. 또한 언어와 문화 차이로 인해 현지 사람들과 자연스럽게 소통하기 어려운 경우도 있습니다.

여기에 더해, 현실에서는 **카드 수수료 문제로 상인들이 현금 결제를 선호**하고, 일부에서는 외국인 관광객에게 바가지를 씌우는 사례도 발생하고 있습니다. 이런 불신의 구조를 끊어내고, 상인과 소비자 모두가 신뢰할 수 있는 결제 환경을 만들고자 **엽전 시스템**을 설계하였습니다. 이를 통해 외국인도 언어적 장벽 없이 편리하게 결제할 수 있는 환경을 구현하고자 합니다.

저희는 이런 문제들을 해결하기 위해, 관광지와 전통시장 정보를 한곳에서 확인하고, 위치를 기반으로 가까운 장소를 추천 받으며, 함께 여행할 사람과 연결될 수 있는 서비스를 기획했습니다.

### 핵심 Pain Point 요약

| # | 대상 | 문제 상황 | 해결 방향 |
| --- | --- | --- | --- |
| P1 | 외국인 관광객+국내 여행자 | 관광지·시장 정보를 한 곳에서 찾기 어렵다 | 위치 기반 통합 탐색 |
| P2 | 외국인 관광객 | 전통 시장에서 언어 소통이 안 돼 결제가 불편하다 | 엽전 시스템 + 번역 기능 |
| P3 | 외국인 관광객 | 바가지 요금이 두렵다 | 상인 인증 마크 + 표준 가격 결제 |
| P4 | 국내 여행자+외국인 관광객 | 혼자 여행하기 불안하고, 동행을 구하기 어렵다 | 커뮤니티 + 채팅 동행 매칭 + 번역 기능 |
| P5 | 전통시장 상인 | 카드 수수료 부담, 현금 선호 구조 | 엽전 기반 수수료 절감 결제 |
| P6 | 모든 사용자 | 이미 가본 곳 말고 새로운 곳을 찾고 싶다 | 방문 이력 기반 개인화 추천 (추후 확장) |

## 4. 프로젝트 목표

저희 프로젝트의 목표는 **여행을 더 편리하고, 더 기억에 남는 경험으로 만드는 것**입니다.

**외국인 관광객**에게는 한국 관광지와 전통시장을 더 쉽게 찾고 즐길 수 있는 환경을 제공하고자 합니다. 한국 여행 중 필요한 위치 정보, 길찾기, 축제 일정, 커뮤니티, 채팅 기능을 통해 여행 과정에서의 불편함을 줄이는 것이 목표입니다.

**한국인 사용자**에게는 외국인과 함께 투어를 하며 자연스럽게 소통할 수 있는 기회를 제공하고자 합니다. 해외에 나가지 않아도 다양한 문화와 사람들을 경험하며, 더 넓은 시야를 가질 수 있도록 돕는 것이 저희 서비스의 또 다른 목적입니다.

추후 확장 단계에서는 Redis 기반 방문 이력 데이터를 활용하여, 사용자가 이미 다녀간 곳이 아닌 **아직 경험하지 못한 새로운 장소와 맛집을 우선 추천**하는 개인화된 여행 경험을 제공하는 것을 목표로 합니다.

## 5. 서비스 방향

저희 서비스는 단순한 관광 정보 제공 사이트가 아닙니다.

사용자가 현재 위치를 기준으로 주변 관광지와 전통시장을 탐색하고, 관심 있는 장소의 상세 정보를 확인한 뒤 길찾기를 통해 이동할 수 있습니다. 또한 캘린더를 통해 축제나 행사 정보를 확인하고, 단체 채팅방을 통해 함께 여행할 사람들과 소통할 수 있습니다.

더 나아가 상인은 자신의 가게 정보와 메뉴를 직접 관리할 수 있고, 관리자는 유저·상인·신고 내역·관광지·축제 정보·광고 등을 관리할 수 있도록 구성하여 **실제 운영 가능한 플랫폼 구조**를 목표로 하고 있습니다.

**MVP 포함 기능**

### 7.3 페이지 / 권한 구조

춘배투어의 화면은 **일반 사용자 페이지 / 상인 페이지 / 관리자 페이지**로 분리한다. 로그인은 가능하면 통합 로그인으로 처리하고, API 접근 권한은 `USER`, `MERCHANT`, `ADMIN` Role 기반으로 제한한다.

**상인 권한 흐름**

```
일반 회원가입
→ USER 계정 생성
→ 상인 등록 신청
→ 관리자 승인
→ role이 MERCHANT로 변경
→ 상인 페이지 접근 가능
```

**Role별 기능 범위**

| Role | 기능 범위 |
| --- | --- |
| USER | 일반 사용자 기능, 프로필, 찜, 리뷰, 결제 |
| MERCHANT | USER 기능 포함, 내 가게 관리, 메뉴 관리, 광고 신청, 정산 신청 |
| ADMIN | 관리자 대시보드, 유저 관리, 상인 승인/거절, 가게 관리, 신고 처리, 광고/정산/인증 심사 |

상인 본인용 API와 관리자 관리용 API는 분리한다.

```
상인 본인용: GET /api/v1/merchants/me/shop
관리자 관리용: GET /api/v1/admin/shops/{shopId}
관리자 관리용: PATCH /api/v1/admin/shops/{shopId}
```
