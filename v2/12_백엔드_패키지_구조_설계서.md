## 0. 최종 패키지 구조 보강

현재 백엔드 도메인 패키지는 다음 21개 기준이다.

| 패키지 | 역할 |
| --- | --- |
| admin | 관리자 대시보드, 감사 로그, 운영 관리 |
| auth | 인증, OAuth, JWT, 보안 필터, 프로필 |
| banner | 배너 |
| chat | 채팅방, 메시지, 참여 신청, WebSocket |
| common | 공통 응답, 예외, 설정, Redis/S3/rate limit |
| community | 자유/동행 게시글, 댓글, 이미지 |
| companionreview | 동행 리뷰, 동행 점수 |
| cs | FAQ, 상담방, 상담 메시지/파일 |
| festival | 축제, 캘린더, 축제 찜, 관리자 수집 |
| like | 공통 찜, target validator |
| market | 전통시장 조회/상세/수집/찜 |
| merchant | 상인 신청, 관리자 승인/거절 |
| notification | 알림 |
| payment | 충전, 환불, QR 결제, PortOne 웹훅 |
| place | 관광지, 지도, 추천, 리뷰, 통계, 카카오 연동 |
| report | 신고, 제재 |
| search | 통합 검색, 장소/축제 검색, 인기/최근/자동완성/오타 교정 |
| shop | 가게, 메뉴, 공지, 이미지, 정산, 광고, QR |
| store | 상품, 주문, 보유 아이템 |
| translation | 번역, 번역 캐시 |
| yeopjeon | 엽전 잔액/거래 내역 |

### 구조 유지 원칙

- 신규 기능은 먼저 담당 도메인 패키지 안에 배치한다.
- 여러 도메인이 반드시 공유하는 응답/예외/설정만 `common`으로 올린다.
- 도메인 간 참조는 가능하면 ID 기반 논리참조 또는 validator/service를 통해 캡슐화한다.
- 외부 API client와 storage 처리는 도메인 서비스에서 직접 흩뿌리지 않고 전용 컴포넌트로 감싼다.

기존 상세 패키지 설명은 아래 본문을 유지하되, 위 표를 최종 실제 코드 기준으로 본다.

## 1. 문서 목적

본 문서는 춘배투어 백엔드 애플리케이션의 패키지 구조를 정의한다.

프로젝트는 Java 21, Spring Boot 4.0.6, Spring Data JPA, QueryDSL, Redis, WebSocket/STOMP, JWT, MySQL 기반으로 구현하며, 기능별 책임이 명확하게 분리된 **도메인 중심 패키지 구조**를 따른다.

이 문서는 다음을 목표로 한다.

- 팀원별 담당 도메인과 실제 코드 패키지의 일관성 확보
- Controller / Service / Repository / Entity / DTO 책임 분리
- 공통 응답, 공통 예외, 보안, 설정 코드의 위치 통일
- REST API, WebSocket, Redis, 외부 API 연동 코드의 관리 기준 정의
- 추후 기능 확장 시 패키지 구조가 무너지지 않도록 기준 제공

## 2. 패키지 구조 설계 원칙

### 2.1 기본 방향

춘배투어 백엔드는 **도메인 중심 구조**를 기본으로 한다.

```
com.chunbae.tour
├── global          # 전역 공통 설정, 응답, 예외, 보안, 외부 인프라 설정
├── domain          # 실제 비즈니스 도메인
└── ChunbaeTourApplication.java
```

도메인별로 `controller`, `service`, `repository`, `entity`, `dto`를 내부에 배치한다.

```
domain/{domainName}
├── controller
├── service
├── repository
├── entity
├── dto
└── type
```

### 2.2 계층 책임

| 계층 | 역할 | 작성 기준 |
| --- | --- | --- |
| Controller | HTTP 요청/응답 처리 | 비즈니스 로직 작성 금지, Service 호출만 수행 |
| WebSocket Controller | STOMP 메시지 수신 처리 | 메시지 인증/권한 확인 후 Service 위임 |
| Service | 핵심 비즈니스 로직 | 트랜잭션, 도메인 규칙, 외부 연동 조합 처리 |
| Repository | DB 접근 | Spring Data JPA, QueryDSL 사용 |
| Entity | DB 테이블 매핑 | 비즈니스 상태 변경 메서드 포함 가능 |
| DTO | 요청/응답 객체 | Entity 직접 노출 금지 |
| Type | Enum 관리 | 상태값, 타입값 등 enum 분리 |
| Global | 공통 관심사 | 보안, 예외, 응답, 설정, 유틸 |

## 4. Global 패키지 상세

### 4.1 global.config

Spring 설정 클래스를 관리한다.

| 클래스 | 역할 |
| --- | --- |
| `JpaConfig` | JPA Auditing 설정 |
| `QueryDslConfig` | JPAQueryFactory Bean 등록 |
| `RedisConfig` | RedisTemplate, Redis 연결 설정 |
| `RedissonConfig` | RedissonClient 및 분산 락 설정 |
| `WebSocketConfig` | STOMP endpoint, pub/sub prefix 설정 |
| `CorsConfig` | 환경별 CORS 정책 설정 |
| `AsyncConfig` | 비동기 처리 설정 |

---

### 4.2 global.security

JWT 인증과 권한 처리를 담당한다.

```text
global/security
├── SecurityConfig.java
├── JwtAuthenticationFilter.java
├── JwtTokenProvider.java
├── CustomUserDetails.java
├── CustomUserDetailsService.java
└── handler
    ├── JwtAuthenticationEntryPoint.java
    └── JwtAccessDeniedHandler.java
```

**설계 기준**

- Access Token은 요청 Header의 `Authorization: Bearer {token}`으로 전달한다.
- Refresh Token은 HttpOnly Cookie로 전달한다.
- 로그아웃된 Access Token은 Redis 블랙리스트로 관리한다.
- API 권한은 `USER`, `MERCHANT`, `ADMIN` Role 기준으로 제한한다.
- WebSocket 연결 시에도 JWT 인증을 수행한다.

---

### 4.3 global.response

모든 REST API 응답은 `ApiResponse<T>`로 감싼다.

```java
@Getter
@AllArgsConstructor
public class ApiResponse<T> {
    private boolean success;
    private String code;
    private String message;
    private T data;

    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>(true, "SUCCESS", "요청이 성공적으로 처리되었습니다.", data);
    }

    public static <T> ApiResponse<T> error(ErrorCode errorCode) {
        return new ApiResponse<>(false, errorCode.getCode(), errorCode.getMessage(), null);
    }
}
```

---

### 4.4 global.exception

공통 예외와 에러코드를 관리한다.

```text
global/exception
├── BusinessException.java
├── ErrorCode.java
└── GlobalExceptionHandler.java
```

**설계 기준**

- 도메인별 예외도 최종적으로는 `BusinessException`을 사용한다.
- 에러코드는 `{도메인 prefix}_{세자리 숫자}` 형식을 따른다.
- 예외 응답은 항상 `ApiResponse.error(errorCode)` 형태로 반환한다.
- 외부 API 장애, AI API 장애, 번역 실패, 파일 업로드 실패는 필요 시 `common_error_logs`에 기록한다.

### 4.5 global.entity

공통 엔티티 필드를 관리한다.

```java
@Getter
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {

    @CreatedDate
    @Column(nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @LastModifiedDate
    @Column(nullable = false)
    private LocalDateTime updatedAt;
}
```

**적용 기준**

- `created_at`, `updated_at`을 사용하는 엔티티는 `BaseEntity`를 상속한다.
- `deleted_at`이 필요한 Soft Delete 엔티티는 각 도메인 Entity에서 별도 관리한다.

## 8. Entity 작성 규칙

### 8.1 기본 규칙

- Entity는 DB 테이블과 직접 매핑한다.
- Setter는 지양하고, 상태 변경 메서드를 작성한다.
- 상태값은 String이 아니라 Enum으로 관리한다.
- 생성/수정 시간은 `BaseEntity`를 상속해 관리한다.
- Soft Delete가 필요한 경우 `deletedAt`을 도메인 Entity에 둔다.

### 8.2 예시

```java
@Entity
@Table(name = "chat_rooms")
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class ChatRoom extends BaseEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private Long postId;

    @Column(nullable = false)
    private Long ownerId;

    @Column(nullable = false, length = 50)
    private String title;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    private ChatRoomStatus status;

    public void close() {
        this.status = ChatRoomStatus.CLOSED;
    }
}
```

## 14. 패키지 작성 시 주의사항

### 14.1 금지 사항

- Controller에서 Repository 직접 호출 금지
- Controller에서 Entity 직접 반환 금지
- 도메인 Service에서 다른 도메인 Repository 직접 남용 금지
- 외부 API 호출 코드를 Controller에 작성 금지
- ErrorCode 문자열을 코드 곳곳에 직접 하드코딩 금지
- WebSocket 메시지에서 클라이언트가 보낸 `senderId`를 그대로 신뢰 금지

### 14.2 권장 사항

- 도메인 간 호출이 필요하면 Service 메서드 또는 Facade 계층으로 분리한다.
- 복잡한 도메인 조합은 `application` 또는 `facade` 패키지 추가를 검토한다.
- 공통 응답은 항상 `ApiResponse<T>`를 사용한다.
- 에러는 항상 `BusinessException(ErrorCode)`로 처리한다.
- Redis Key는 문자열 하드코딩 대신 상수 또는 Prefix 클래스로 관리한다.

## 17. 결론

춘배투어 백엔드는 도메인 중심 패키지 구조를 사용한다.

`global` 패키지는 공통 설정, 보안, 예외, 응답, 외부 인프라 연동을 담당하고, `domain` 패키지는 실제 비즈니스 기능을 담당한다.

이 구조를 따르면 팀원별 담당 영역이 명확해지고, API 명세서·ERD·기능 명세서와 실제 코드 구조를 일관되게 연결할 수 있다.
