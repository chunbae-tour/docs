## 0. 최종 동시성 설계 보강

최종 구현에서는 모든 기능에 같은 락을 쓰지 않고, 데이터 성격에 따라 제어 방식을 구분한다.

| 영역 | 동시성 위험 | 최종 방어 방식 |
| --- | --- | --- |
| 관광지 상세 캐시 | 캐시 만료 시 DB 동시 조회 폭증 | Redis/Redisson 기반 cache stampede 방어 |
| 관광지 조회수/좋아요 | 요청마다 DB update 시 병목과 충돌 | Redis counter 선반영 + dirty marker + chunk sync |
| 인기 검색어 | 다중 사용자 동시 검색 집계 | Redis ZSet 원자 증가, 자정 스냅샷 |
| 최근 검색어 | 사용자별 중복/순서 관리 | Redis 사용자별 key 분리 |
| QR 결제 | 중복 결제/중복 승인/만료 처리 | idempotency key, 상태 전이 검증, 트랜잭션 |
| 채팅방 참여 | 정원 초과, 중복 신청, 방장 위임 | 상태 기반 검증, 멤버 상태 관리 |
| 파일 업로드 | S3 성공 후 DB 실패 | orphan cleanup, object key 기반 보상 처리 |
| Redis Cluster | CROSSSLOT 오류 | hash tag와 개별 GET/pipeline 전략 구분 |

### 관광지 통계 write-behind 핵심

1. 사용자 요청에서 Redis counter를 먼저 증가/감소한다.
2. 변경된 placeId를 dirty set에 남긴다.
3. 스케줄러가 dirty set을 scan한다.
4. chunk 단위 트랜잭션으로 DB에 반영한다.
5. cleanup 전 Redis 값이 바뀌었는지 재확인한다.
6. 값이 바뀌었으면 dirty marker를 유지해 다음 주기에 재처리한다.

### 검색 집계 정책

- 일반 검색은 인기 검색어에 반영한다.
- 내부 선택용 검색은 `track=false` 또는 비집계 `source`로 제외한다.
- 통합 검색과 장소/축제 검색의 집계 정책은 같은 기준을 사용한다.

## 1. 개요

춘배투어는 다수의 사용자가 동시에 접근하는 상황에서 **데이터 정합성**을 보장해야 하는 시나리오가 여러 도메인에 걸쳐 존재한다.

본 문서는 각 시나리오별 **동시성 문제 정의 → 선택한 전략 → 구현 방식 → 실패 처리**를 명세한다.

### 1.1 동시성 문제가 발생하는 도메인

| 도메인 | 시나리오 | 문제 유형 |
| --- | --- | --- |
| 스토어/결제 | 한정 수량 상품 구매 | 재고 초과 차감 (Race Condition) |
| 스토어/결제 | 엽전 차감 | 잔액 음수 발생 (Lost Update) |
| 커뮤니티 | 좋아요 / 조회수 | 카운트 유실 (Lost Update) |
| 관광지/검색 | 인기 검색어 집계 | 카운트 유실 (Lost Update) |
| 채팅/매칭 | 채팅방 인원 초과 참여 | 정원 초과 (Race Condition) |

## 2. 전략 선택 기준

동시성 제어 전략은 **충돌 빈도**와 **데이터 정합성 요구 수준**을 기준으로 선택한다.

| 전략 | 설명 | 적합한 상황 |
| --- | --- | --- |
| **Redis 분산 락** | Redisson 기반 분산 환경 락 | 충돌 빈도 높음 + 정합성 필수 |
| **DB 비관적 락** | `SELECT FOR UPDATE` | 최종 정합성 보장이 필수인 금전 처리 |
| **DB 낙관적 락** | `@Version` 버전 관리 | 충돌 빈도 낮음 + 성능 우선 |
| **Redis Atomic 연산** | `INCR`, `DECR`, `ZINCRBY` | 카운트 집계, 순위 등 단순 증감 |

### Trade-off 비교

| 항목 | Redis 분산 락 | DB 비관적 락 | DB 낙관적 락 | Redis Atomic |
| --- | --- | --- | --- | --- |
| 성능 | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 정합성 보장 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| 구현 복잡도 | 중간 | 낮음 | 낮음 | 낮음 |
| 분산 환경 적합성 | ✅ | ❌ (단일 DB) | ⚠️ 재시도 로직 필요 | ✅ |
| 데드락 위험 | 낮음 | 있음 | 없음 | 없음 |

## 3. 시나리오별 동시성 제어 설계

---

### 시나리오 1 — 한정 수량 상품 구매 (재고 초과 차감 방지) ⭐

> **담당**: 신현민 | **위험도**: 🔴 높음
> 

### 3.1.1 문제 정의

한정 수량 상품(예: 재고 1개)에 사용자 A, B가 동시에 구매 요청을 보낼 경우,
두 요청 모두 재고 확인 시점에 `재고 = 1`을 읽고 동시에 차감하면 **재고가 -1**이 되는 문제 발생.

```
[문제 상황]
재고: 1개

User A: SELECT stock → 1 (구매 가능)
User B: SELECT stock → 1 (구매 가능)  ← 동시에 읽음
User A: UPDATE stock = stock - 1 → 0
User B: UPDATE stock = stock - 1 → -1  ← 재고 음수 발생!
```

### 3.1.2 해결 전략 — 3단계 보호 전략

```
구매 요청
    │
    ├─ [1단계] Redis 가점유 재고 -1 (DECR)
    │   재고 0 미만 → 즉시 품절 안내 + 가점유 복구 (INCR)
    │
    ├─ [2단계] Redis 분산 락 획득 (사용자 ID 기준)
    │   락 획득 실패 → "잠시 후 다시 시도해주세요" 안내
    │
    ├─ [3단계] DB 비관적 락으로 재고 최종 확인 (SELECT FOR UPDATE)
    │   재고 없음 → 가점유 복구 + 락 해제
    │
    ├─ PG사 결제 or 엽전 차감
    │
    ├─ 성공 → DB 재고 확정 + 락 해제
    └─ 실패 → 가점유 복구 + 락 해제 + 트랜잭션 롤백
```

### 3.1.3 각 단계별 역할 및 이유

| 단계 | 기술 | 역할 | 선택 이유 |
| --- | --- | --- | --- |
| 1단계 | Redis DECR | 빠른 선점 필터링 | DB 접근 전 Redis에서 대부분의 품절 요청을 차단 → DB 부하 감소 |
| 2단계 | Redis 분산 락 | 동일 사용자 중복 요청 방지 | 사용자 ID 기준 락으로 같은 사용자의 동시 요청 차단 |
| 3단계 | DB 비관적 락 | 최종 정합성 보장 | Redis 장애 시에도 DB 레벨에서 재고 정합성 보장 |

### 3.1.4 구현 코드 시나리오

```java
// 1단계: Redis 가점유
Long remaining = redisTemplate.opsForValue().decrement("stock:" + productId);
if (remaining < 0) {
    redisTemplate.opsForValue().increment("stock:" + productId); // 복구
    throw new OutOfStockException();
}

// 2단계: 분산 락
RLock lock = redissonClient.getLock("purchase:lock:" + userId);
try {
    if (!lock.tryLock(3, 5, TimeUnit.SECONDS)) {
        redisTemplate.opsForValue().increment("stock:" + productId); // 복구
        throw new LockAcquisitionFailException();
    }

    // 3단계: DB 비관적 락
    Product product = productRepository.findByIdWithLock(productId); // SELECT FOR UPDATE
    if (product.getStock() <= 0) {
        redisTemplate.opsForValue().increment("stock:" + productId); // 복구
        throw new OutOfStockException();
    }

    // 결제 처리 (PG or 엽전)
    paymentService.process(order);

    // 재고 확정
    product.decreaseStock();

} finally {
    lock.unlock();
}
```

### 3.1.5 실패 시나리오 매트릭스

| 실패 지점 | 처리 방법 | Redis 복구 | DB 롤백 |
| --- | --- | --- | --- |
| 1단계 재고 없음 | 품절 안내 | INCR 복구 | 불필요 |
| 2단계 락 실패 | 재시도 안내 | INCR 복구 | 불필요 |
| 3단계 DB 재고 없음 | 품절 안내 | INCR 복구 | 불필요 |
| 결제 실패 | 결제 실패 안내 | INCR 복구 | 트랜잭션 롤백 |
| 결제 후 DB 저장 실패 | 결제 취소 요청 | INCR 복구 | 트랜잭션 롤백 |

### 시나리오 2 — 엽전 차감 (잔액 음수 방지)

> **담당**: 신현민 | **위험도**: 🔴 높음
> 

### 3.2.1 문제 정의

사용자가 동시에 여러 건의 엽전 결제를 요청할 경우,
잔액 확인 시점과 차감 시점 사이에 다른 요청이 끼어들어 **잔액이 음수**가 되는 문제 발생.

```
[문제 상황]
잔액: 1,000 엽전

요청 A: SELECT balance → 1,000 (결제 가능)
요청 B: SELECT balance → 1,000 (결제 가능)  ← 동시에 읽음
요청 A: UPDATE balance = 1000 - 800 → 200
요청 B: UPDATE balance = 1000 - 800 → 200  ← 잔액 음수 발생! (실제 -600)
```

### 3.2.2 해결 전략

```
엽전 결제 요청
    │
    ├─ Redis 분산 락 획득 (사용자 ID 기준)
    │   락 획득 실패 → "처리 중입니다" 안내
    │
    ├─ DB 비관적 락으로 잔액 조회 (SELECT FOR UPDATE)
    │
    ├─ 잔액 검증 (balance >= amount)
    │   잔액 부족 → 락 해제 + 충전 유도 안내
    │
    ├─ 잔액 차감 + 결제 내역 기록
    │
    ├─ 성공 → 락 해제 + 커밋
    └─ 실패 → 락 해제 + 롤백
```

### 3.2.3 실패 시나리오 매트릭스

| 실패 지점 | 처리 방법 | 정합성 보장 |
| --- | --- | --- |
| 락 획득 실패 | 재시도 안내 (최대 3회) | ✅ |
| 잔액 부족 | 충전 유도 안내 | ✅ |
| DB 저장 실패 | 트랜잭션 롤백 | ✅ |
| 서버 장애 (락 보유 중) | TTL 만료 후 자동 락 해제 | ✅ |

### 시나리오 3 — 채팅방 인원 초과 참여 방지

> 채팅방 상태는 `OPEN / FULL / CLOSED`로 관리한다. 참여자 상태는 `member_state` 하나로 관리하며 `OWNER_ACTIVE / MEMBER_ACTIVE / MEMBER_LEFT / MEMBER_KICKED`를 사용한다.
> 

> **담당**: 임하은 | **위험도**: 🟡 중간
> 

### 3.3.1 문제 정의

최대 인원이 설정된 채팅방(예: 최대 5명)에 동시에 여러 사용자가 참여 요청을 보낼 경우,
정원을 초과하여 참여가 허용되는 문제 발생.

```
[문제 상황]
채팅방 현재 인원: 4명 / 최대: 5명 (잔여 1명)

User A: SELECT count → 4 (참여 가능)
User B: SELECT count → 4 (참여 가능)  ← 동시에 읽음
User A: INSERT 참여 → 5명
User B: INSERT 참여 → 6명  ← 정원 초과!
```

### 3.3.2 해결 전략

```
채팅방 참여 승인 요청
    │
    ├─ Redis 분산 락 획득 (chatRoomId 기준)
    │   락 획득 실패 → "잠시 후 다시 시도해주세요" 안내
    │
    ├─ DB 현재 인원 조회 (SELECT FOR UPDATE)
    │
    ├─ 정원 검증 (currentCount < maxCount)
    │   정원 초과 → 락 해제 + "채팅방이 가득 찼습니다" 안내
    │
    ├─ 참여 처리 (`member_state = MEMBER_ACTIVE`)
    │
    ├─ 성공 → 락 해제 + WebSocket 입장 알림
    └─ 실패 → 락 해제 + 롤백
```

### 3.3.3 실패 시나리오 매트릭스

| 실패 지점 | 처리 방법 | 정합성 보장 |
| --- | --- | --- |
| 락 획득 실패 | 재시도 안내 | ✅ |
| 정원 초과 | “방이 가득 찼습니다” 안내 | ✅ |
| 이미 참여한 사용자 중복 요청 | “이미 참여 중입니다” 안내 | ✅ |
| DB 저장 실패 | 트랜잭션 롤백 | ✅ |

### 시나리오 4 — 인기 검색어 집계 (카운트 유실 방지)

> **담당**: 김인목 | **위험도**: 🟢 낮음
> 

### 3.4.1 문제 정의

다수의 사용자가 동시에 검색할 경우, DB UPDATE로 카운트를 증가시키면
동시 요청 간 **Lost Update**가 발생하여 일부 카운트가 유실될 수 있다.

### 3.4.2 해결 전략 — Redis Atomic 연산

인기 검색어는 **완벽한 정합성보다 성능이 중요**한 영역이므로,
Redis ZSet의 `ZINCRBY` 원자적 연산으로 처리한다.

```
검색 요청
    │
    ├─ Redis ZINCRBY "search:ranking" 1 "{keyword}"  ← Atomic 연산
    │   (동시 요청도 Redis 단일 스레드로 순차 처리됨)
    │
    └─ 검색 결과 반환

[별도 스케줄러 - 주기적 실행]
    │
    └─ Redis ZSet TOP 10 → DB 동기화 (선택적)
```

### 3.4.3 Redis ZSet 선택 이유

| 항목 | DB UPDATE | Redis ZINCRBY |
| --- | --- | --- |
| 동시성 처리 | 락 필요 | 불필요 (단일 스레드) |
| 성능 | 낮음 | 매우 높음 |
| 정합성 | 완벽 | 근사치 (Redis 장애 시 유실 가능) |
| 순위 정렬 | ORDER BY 필요 | 자동 정렬 (ZSet 특성) |
| 적합 여부 | ❌ | ✅ |

> **근거**: 인기 검색어는 1~2건 유실이 서비스 품질에 영향을 주지 않는다.
반면 실시간 응답성이 중요하므로 Redis Atomic 연산이 최적.
>

### 시나리오 5 — 게시글 좋아요 / 관광지 조회수

> **담당**: 박경화 | **위험도**: 🟢 낮음
> 

### 3.5.1 해결 전략 — Redis INCR + 배치 동기화

좋아요·조회수는 인기 검색어와 동일하게 **성능 우선** 영역이다.

```
좋아요 / 조회수 요청
    │
    ├─ Redis INCR "post:like:{postId}" or "place:view:{placeId}"
    │   (Atomic 연산으로 동시성 문제 없음)
    │
    └─ 응답 반환 (즉시)

[배치 스케줄러 - 5분 주기]
    │
    └─ Redis 카운트 → DB UPDATE (벌크 처리)
```

### 3.5.2 좋아요 중복 방지

좋아요는 카운트뿐 아니라 **중복 방지**도 필요하다.

```
좋아요 요청
    │
    ├─ Redis SET "post:liked:{postId}:{userId}" 존재 여부 확인
    │   이미 존재 → "이미 좋아요를 눌렀습니다" 안내
    │
    ├─ Redis SET "post:liked:{postId}:{userId}" = 1 (SET NX)
    │   ← SETNX로 중복 방지 (Atomic)
    │
    └─ Redis INCR "post:like:{postId}"
```

## 4. 전체 동시성 제어 시나리오 매트릭스

| 시나리오 | 도메인 | 전략 | 락 단위 | 위험도 | 담당 |
| --- | --- | --- | --- | --- | --- |
| 한정 수량 상품 구매 | 스토어 | Redis 가점유 + 분산 락 + DB 비관적 락 | 사용자 ID | 🔴 높음 | 신현민 |
| 엽전 차감 | 결제 | Redis 분산 락 + DB 비관적 락 | 사용자 ID | 🔴 높음 | 신현민 |
| 채팅방 참여 | 채팅 | Redis 분산 락 + DB 비관적 락 | 채팅방 ID | 🟡 중간 | 임하은 |
| 인기 검색어 집계 | 검색 | Redis ZINCRBY (Atomic) | - | 🟢 낮음 | 김인목 |
| 좋아요 / 조회수 | 커뮤니티 | Redis INCR + SETNX (Atomic) | - | 🟢 낮음 | 박경화 |

## 5. 공통 분산 락 구현 가이드

### 5.1 락 키 네이밍 규칙

```
{도메인}:lock:{식별자}

예시)
purchase:lock:{userId}         ← 상품 구매
payment:lock:{userId}          ← 엽전 결제
chatroom:lock:{chatRoomId}     ← 채팅방 참여
```

### 5.2 분산 락 공통 설정값

| 항목 | 값 | 이유 |
| --- | --- | --- |
| **waitTime** | 3초 | 3초 내 락 획득 못하면 실패 처리 |
| **leaseTime** | 5초 | 로직 실행 중 장애 시 5초 후 자동 해제 |
| **재시도** | 최대 3회 | 일시적 충돌은 재시도로 해결 |

```java
// 공통 분산 락 유틸
@Component
public class DistributedLockExecutor {

    private final RedissonClient redissonClient;

    public <T> T execute(String lockKey, long waitTime, long leaseTime,
                         TimeUnit timeUnit, Supplier<T> supplier) {
        RLock lock = redissonClient.getLock(lockKey);
        try {
            if (!lock.tryLock(waitTime, leaseTime, timeUnit)) {
                throw new LockAcquisitionFailException("락 획득 실패: " + lockKey);
            }
            return supplier.get();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new LockInterruptedException(e);
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

### 5.3 DB 비관적 락 Repository 패턴

```java
// 재고 조회 (SELECT FOR UPDATE)
public interface ProductRepository extends JpaRepository<Product, Long> {
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdWithLock(@Param("id") Long id);
}

// 엽전 잔액 조회 (SELECT FOR UPDATE)
public interface WalletRepository extends JpaRepository<Wallet, Long> {
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT w FROM Wallet w WHERE w.userId = :userId")
    Optional<Wallet> findByUserIdWithLock(@Param("userId") Long userId);
}
```

## 6. 장애 대응 시나리오

### 6.1 Redis 장애 시 Fallback 전략

| 시나리오 | Redis 역할 | 장애 시 처리 |
| --- | --- | --- |
| 한정 수량 구매 | 1단계 가점유 | Redis 없이 DB 비관적 락만으로 처리 (성능 저하 감수) |
| 엽전 결제 | 분산 락 | DB 비관적 락만으로 처리 |
| 인기 검색어 | 집계 저장소 | 임시 메모리 집계 or 집계 기능 일시 중단 |
| 채팅방 참여 | 분산 락 | DB 비관적 락만으로 처리 |

> ⚠️ **AWS ElastiCache Redis는 Multi-AZ 구성으로 장애 내구성을 확보한다.**
상세 내용은 인프라 아키텍처 문서 참조.
> 

### 6.2 데드락 방지 전략

- 분산 락은 항상 **leaseTime(TTL)**을 설정하여 장애 시 자동 해제 보장
- DB 비관적 락은 **트랜잭션 타임아웃**을 설정 (기본 30초)
- 락 획득 순서는 항상 **상위 리소스 → 하위 리소스** 순으로 고정 (순환 참조 방지)

### 시나리오 6 — QR 결제 상태 전이 중복 처리 방지

> QR 결제 상태는 `PENDING_CONFIRM / COMPLETED / REJECTED / EXPIRED / CANCELLED`로 관리한다.
> 

### 3.6.1 문제 정의

QR 결제 요청이 생성된 뒤, 상인 승인/거절, 사용자 취소, 만료 스케줄러가 거의 동시에 실행되면 이미 처리된 결제가 다시 처리될 수 있다.

```
PENDING_CONFIRM 상태의 QR 결제 요청

사용자: 결제 요청 취소
상인: 거의 동시에 결제 승인
스케줄러: 제한 시간 만료 처리

→ 하나의 요청에 대해 CANCELLED / COMPLETED / EXPIRED가 중복 반영될 위험
```

### 3.6.2 해결 전략

```
QR 결제 상태 변경 요청
    │
    ├─ Redis 분산 락 획득 (qrPayRequestId 기준)
    │
    ├─ DB에서 QR 결제 요청 조회 (SELECT FOR UPDATE)
    │
    ├─ 현재 상태 검증
    │   PENDING_CONFIRM 상태에서만 COMPLETED / REJECTED / EXPIRED / CANCELLED 변경 가능
    │
    ├─ 이미 처리된 상태면 재처리 차단
    │
    └─ 상태 변경 + 잔액 확정/복구 처리
```

### 3.6.3 상태 전이 규칙

| 현재 상태 | 변경 가능 상태 | 처리 주체 |
| --- | --- | --- |
| PENDING_CONFIRM | COMPLETED | 상인 승인 |
| PENDING_CONFIRM | REJECTED | 상인 거절 |
| PENDING_CONFIRM | EXPIRED | 만료 스케줄러 |
| PENDING_CONFIRM | CANCELLED | 사용자 직접 취소 |
| COMPLETED / REJECTED / EXPIRED / CANCELLED | 변경 불가 | 재처리 차단 |
