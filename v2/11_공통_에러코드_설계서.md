## 0. 최종 에러 코드 운용 기준

에러 코드는 `ErrorCode.java`가 단일 출처이다. API 문서에는 주요 실패 케이스를 함께 적고, 프론트는 `code` 문자열을 기준으로 분기한다.

| Prefix | 담당 도메인 |
| --- | --- |
| COMMON | 공통, validation, cursor, rate limit, 외부 서비스 |
| AUTH | 인증, 권한, OAuth, 사용자, 제재 |
| PLACE / FESTIVAL | 관광지, 전통시장, 축제, 검색 일부, 찜/리뷰 일부 |
| COMMUNITY | 게시글, 댓글, 커뮤니티 이미지 |
| CHAT / CR | 채팅, 참여 신청, 동행, 동행 리뷰 |
| CS / FAQ | 고객센터 상담, FAQ |
| REPORT | 신고, 관리자 처리, 제재 이력 |
| NOTIFICATION | 알림 |
| MERCHANT / SHOP | 상인 신청, 가게, 메뉴, 정산, 광고, 이미지 |
| PAY / STORE | 결제, 환불, QR, 스토어, 아이템 |

### 공통 응답 원칙

- 비즈니스 오류는 `BusinessException(ErrorCode)`로 표현한다.
- validation 오류는 `GlobalExceptionHandler`에서 `COMMON_002` 또는 세부 공통 코드로 변환한다.
- 서버 내부 상세 원인은 사용자 응답에 노출하지 않고 로그에 남긴다.
- 기존 프론트가 사용하는 code 문자열은 호환성 검토 없이 변경하지 않는다.

기존 상세 에러 표는 아래 본문을 유지한다.

## 1. 에러 응답 공통 포맷

모든 API 에러 응답은 아래 형식을 따른다.

```json
{
  "success": false,
  "code": "AUTH_001",
  "message": "이메일 또는 비밀번호가 올바르지 않습니다.",
  "data": null
}
```

### 성공 응답 포맷 (참고)

```json
{
  "success": true,
  "code": "SUCCESS",
  "message": "요청이 성공적으로 처리되었습니다.",
  "data": { ... }
}
```

### 공통 응답 클래스 (Java)

```java
@Getter
@AllArgsConstructor
public class ApiResponse<T> {
    private boolean success;
    private String code;
    private String message;
    private T data;

    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>(true, "SUCCESS", "요청이 성공적으로 처리되었습니다.", data);
    }

    public static <T> ApiResponse<T> error(ErrorCode errorCode) {
        return new ApiResponse<>(false, errorCode.getCode(), errorCode.getMessage(), null);
    }
}
```

## 2. 에러코드 네이밍 규칙

```
{도메인 prefix}_{세자리 숫자}

예시)
AUTH_001   ← 인증/회원 도메인, 001번 에러
PLACE_003  ← 관광지 도메인, 003번 에러
PAY_002    ← 결제 도메인, 002번 에러
```

### 도메인 Prefix 정의

> 용어 기준: MERCHANT = 상인 계정/상인 신청/상인 권한, SHOP = 실제 가게 정보/메뉴/정산, STORE = 일반 사용자가 구매하는 상품/쿠폰 판매 영역
> 

| Prefix | 도메인 | 담당자 |
| --- | --- | --- |
| `AUTH` | 회원 / 인증 | 정민교 |
| `USER` | 마이페이지 / 사용자 | 정민교 |
| `PLACE` | 관광지 / 지도 / 검색 | 김인목 |
| `COMMUNITY` | 커뮤니티 / 리뷰 | 박경화 |
| `CHAT` | 채팅 / 매칭 | 임하은 |
| `PAY` | 결제 / 엽전 | 신현민 |
| `STORE` | 스토어 / 상품 | 신현민 |
| `MERCHANT` | 상인 계정 / 상인 신청 / 상인 권한 | 신현민 |
| `SHOP` | 가게 / 메뉴 / 정산 | 신현민 |
| `ADMIN` | 관리자 | 정민교 |
| `COMMON` | 공통 (서버 오류 등) | 전체 |

## 3. HTTP 상태 코드 매핑 기준

| HTTP 상태 코드 | 의미 | 사용 상황 |
| --- | --- | --- |
| `200 OK` | 성공 | 정상 처리 |
| `201 Created` | 생성 성공 | 리소스 생성 (회원가입, 게시글 작성 등) |
| `400 Bad Request` | 잘못된 요청 | 유효성 검사 실패, 잘못된 입력값 |
| `401 Unauthorized` | 인증 실패 | 토큰 없음, 토큰 만료, 로그인 실패 |
| `403 Forbidden` | 권한 없음 | 접근 권한 부족 (일반 유저가 관리자 API 접근 등) |
| `404 Not Found` | 리소스 없음 | 존재하지 않는 리소스 조회 |
| `409 Conflict` | 충돌 | 중복 데이터, 이미 처리된 요청 |
| `429 Too Many Requests` | 요청 초과 | Rate Limit 초과 |
| `500 Internal Server Error` | 서버 오류 | 예상치 못한 서버 오류 |
| `503 Service Unavailable` | 서비스 불가 | 외부 API 장애, 락 획득 실패 |

### 4.1 공통 (COMMON)

> 번역 API, AI FAQ, PG사, Map API 등 외부 연동 실패는 필요한 경우 `common_error_logs`에 상세 내용을 기록한다. 클라이언트에는 아래 공통 에러 포맷으로 응답한다.
> 

| 에러코드 | HTTP | 메시지 | 발생 상황 |
| --- | --- | --- | --- |
| `COMMON_001` | 500 | 서버 내부 오류가 발생했습니다. | 예상치 못한 서버 오류 |
| `COMMON_002` | 400 | 요청 형식이 올바르지 않습니다. | JSON 파싱 실패, 잘못된 요청 형식 |

| `COMMON_004` | 400 | 입력값이 유효하지 않습니다. | `@Valid` 검증 실패 (형식, 범위 등) |
| `COMMON_005` | 404 | 요청한 리소스를 찾을 수 없습니다. | 존재하지 않는 엔드포인트 |

| `COMMON_007` | 503 | 외부 서비스 연동 중 오류가 발생했습니다. | Map API, 번역 API, PG사 장애 |

## 5. 글로벌 예외 처리 (GlobalExceptionHandler)

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 커스텀 비즈니스 예외
    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ApiResponse<?>> handleBusinessException(BusinessException e) {
        ErrorCode errorCode = e.getErrorCode();
        return ResponseEntity
                .status(errorCode.getHttpStatus())
                .body(ApiResponse.error(errorCode));
    }

    // @Valid 유효성 검사 실패
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ApiResponse<?>> handleValidationException(
            MethodArgumentNotValidException e) {
        String message = e.getBindingResult().getFieldErrors().stream()
                .map(FieldError::getDefaultMessage)
                .findFirst()
                .orElse("입력값이 유효하지 않습니다.");
        return ResponseEntity
                .status(HttpStatus.BAD_REQUEST)
                .body(new ApiResponse<>(false, "COMMON_004", message, null));
    }

    // 인증 실패
    @ExceptionHandler(AccessDeniedException.class)
    public ResponseEntity<ApiResponse<?>> handleAccessDeniedException() {
        return ResponseEntity
                .status(HttpStatus.FORBIDDEN)
                .body(ApiResponse.error(ErrorCode.AUTH_007));
    }

    // 예상치 못한 서버 오류
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ApiResponse<?>> handleException(Exception e) {
        log.error("Unexpected error: ", e);
        return ResponseEntity
                .status(HttpStatus.INTERNAL_SERVER_ERROR)
                .body(ApiResponse.error(ErrorCode.COMMON_001));
    }
}
```

## 7. BusinessException 구조

```java
// 기본 비즈니스 예외
@Getter
public class BusinessException extends RuntimeException {
    private final ErrorCode errorCode;

    public BusinessException(ErrorCode errorCode) {
        super(errorCode.getMessage());
        this.errorCode = errorCode;
    }
}

// 도메인별 예외 (선택적으로 세분화)
public class AuthException extends BusinessException {
    public AuthException(ErrorCode errorCode) {
        super(errorCode);
    }
}

public class PaymentException extends BusinessException {
    public PaymentException(ErrorCode errorCode) {
        super(errorCode);
    }
}

// 사용 예시
if (wallet.getBalance() < amount) {
    throw new PaymentException(ErrorCode.PAY_001);
}
```
